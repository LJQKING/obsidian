**Android Jetpack DataStore** 是 Google 推荐的轻量级数据存储方案，用于取代 **SharedPreferences**。它基于 Kotlin 协程（Coroutines）和 Flow 构建，提供异步、事务性（transactional）和一致性的数据存储。

DataStore 有两种主要实现：

- **Preferences DataStore**：键值对存储（类似 SharedPreferences），无类型安全。
- **Proto DataStore**（或自定义 Serializer，如 JSON）：类型化对象存储，使用 Protocol Buffers 等序列化，确保类型安全。

### 1. 原理（核心机制）

DataStore 的核心优势在于**事务性（ACID 部分保证）**和**线程安全**：

- **Actor 模型并发控制**：内部使用 Kotlin Actor（基于 Channel）序列化所有读写操作，确保原子性（atomic read-modify-write）。避免 SharedPreferences 常见的“丢失更新”问题（如并发自增计数器）。
- **数据持久化**：数据以二进制文件存储（Preferences 为 `.preferences_pb`，Proto 为 `.pb`）。内存中维护缓存，更新时全量序列化写入（不支持部分更新，适合小数据）。
- **Flow 响应式**：`data` 属性返回 `Flow<T>`，数据变化时自动发射新值，支持观察者模式。
- **CoroutineScope**：所有 I/O 操作默认在 `Dispatchers.IO`，避免主线程阻塞。
- **单实例限制**：同一进程同一文件只能有一个 DataStore 实例，否则抛 `IllegalStateException`。使用属性委托（`preferencesDataStore` / `dataStore`）强制单例。
- **错误处理**：支持 `CorruptionHandler` 处理文件损坏（如序列化失败时回滚到默认值）。
- **MultiProcess 支持**：较新版本有 `MultiProcessDataStore`，支持跨进程（需注意文件锁等）。

**与 SharedPreferences 对比**：
- SharedPreferences：XML 存储、同步 API 易 ANR、无事务保证、类型不安全。
- DataStore：异步、安全、事务性、Flow 响应式、Proto 类型安全。

性能上，DataStore 在原子性保证下可能略慢于 MMKV（mmap）或优化后的 SharedPreferences，但一致性更好。

### 2. 使用（代码示例）

**依赖**（最新版本以官方为准，通常 `1.x`）：

```kotlin
// Preferences
implementation("androidx.datastore:datastore-preferences:1.2.1")

// Proto / 自定义
implementation("androidx.datastore:datastore:1.2.1")
```

#### Preferences DataStore 示例

```kotlin
// Top-level（单例）
val Context.settingsDataStore: DataStore<Preferences> by preferencesDataStore(name = "settings")

// Keys
val COUNTER_KEY = intPreferencesKey("counter")
val USERNAME_KEY = stringPreferencesKey("username")

// 读取（Flow）
val counterFlow: Flow<Int> = context.settingsDataStore.data
    .map { preferences -> preferences[COUNTER_KEY] ?: 0 }

// 写入（事务）
suspend fun incrementCounter(context: Context) {
    context.settingsDataStore.edit { preferences ->
        val current = preferences[COUNTER_KEY] ?: 0
        preferences[COUNTER_KEY] = current + 1
    }
}

// 观察（Compose / ViewModel）
viewModelScope.launch {
    dataStore.data.collect { /* 更新 UI */ }
}
```

#### Proto DataStore 示例

1. 定义 `proto`（`app/src/main/proto/settings.proto`）：

```proto
syntax = "proto3";
option java_package = "com.example";
option java_multiple_files = true;

message Settings {
  int32 counter = 1;
  string username = 2;
}
```

2. 生成代码 + Serializer：

```kotlin
object SettingsSerializer : Serializer<Settings> {
    override val defaultValue: Settings = Settings.getDefaultInstance()
    override suspend fun readFrom(input: InputStream): Settings = 
        try { Settings.parseFrom(input) } catch (e: Exception) { 
            throw CorruptionException("...", e) 
        }
    override suspend fun writeTo(t: Settings, output: OutputStream) = t.writeTo(output)
}
```

3. 创建 & 使用：

```kotlin
val Context.protoDataStore: DataStore<Settings> by dataStore(
    fileName = "settings.pb",
    serializer = SettingsSerializer
)

// 更新
suspend fun update() {
    dataStore.updateData { current ->
        current.toBuilder().setCounter(current.counter + 1).build()
    }
}
```

**迁移 SharedPreferences**：使用 `SharedPreferencesMigration`。

**正确使用规则**：
- 顶层声明单例。
- `updateData` / `edit` 是事务性的（要么全成功，要么回滚）。
- 避免在主线程阻塞读（用 `first()` 或 Flow）。
- 数据类型不可变。

### 3. 源码关键分析

核心类（基于开源分析）：

- **SingleProcessDataStore.kt**：核心实现，使用 Actor 处理消息（Read/Write/Update）。维护内存缓存 + 文件 I/O。
- **DataStore 接口**：`val data: Flow<T>` + `suspend fun updateData(transform: suspend (T) -> T): T`。
- **Preferences**：内部类似 Map，使用 `MutablePreferences` 编辑。
- **Serializer**：读写抽象，支持 Proto/JSON 等。
- **Coordinator / File**：处理文件锁定、版本、损坏恢复。
- **Actor 序列化**：所有操作通过 Channel 排队，确保原子性（这是性能 trade-off 的来源）。

源码强调**一致性优先**：全量写入 + 事务，而非高性能部分更新（适合 Room 处理大数据）。

### 4. 面试要点

**常见问题**：

1. **DataStore 比 SharedPreferences 好在哪里？**  
   异步（无 ANR）、事务性（原子更新）、Flow 响应式、类型安全（Proto）、一致性保证。SharedPreferences 易丢失更新、XML 低效、主线程风险。

2. **DataStore 内部如何保证线程安全和原子性？**  
   Actor 模型 + Channel 序列化操作。`updateData` 是 read-modify-write 原子事务。提及 SharedPreferences 的局限。

3. **为什么只能一个实例？多个实例会怎样？**  
   Actor 状态不共享，会破坏一致性。库运行时检测并抛异常。

4. **Preferences vs Proto？**  
   Preferences：简单、无 schema；Proto：类型安全、效率高、不可变。

5. **性能对比？何时用 Room？**  
   DataStore 适合小量简单数据（设置、用户偏好）。大数据/关系型/部分更新 → Room。MMKV 适合高性能键值（但无 Flow 响应式）。

6. **如何处理同步需求？**  
   官方建议预加载或 `runBlocking`（测试用），生产避免。或用 `first()` 获取快照。

7. **迁移策略？**  
   `SharedPreferencesMigration` + 数据一致性处理。

8. **MultiProcess 支持？**  
   新版有，注意跨进程锁和一致性。

**加分点**：提到 Actor 源码、CorruptionHandler、KMP 支持、与 Hilt/ViewModel 结合、Flow 去重（`distinctUntilChanged`）。
