`LiveData` 是 Android Jetpack 中用于数据观察的组件，虽然生命周期感知（Lifecycle Aware）是它最大的优点，但在大型项目（MVVM、Compose、复杂列表、车载项目）中也存在不少缺陷，因此很多新项目逐渐转向 Kotlin Flow。

---

# 一、LiveData 的主要缺点

## 1. 数据倒灌（Sticky Event）

最经典的问题。

### 场景

ViewModel：

```java
MutableLiveData<String> liveData = new MutableLiveData<>();

liveData.setValue("登录成功");
```

Activity：

```java
liveData.observe(this, s -> {
    Toast.makeText(this, s, Toast.LENGTH_SHORT).show();
});
```

旋转屏幕：

```text
Activity销毁
↓
重新创建
↓
重新observe
↓
收到旧数据
↓
Toast再次弹出
```

结果：

```text
登录成功
登录成功
```

这就是：

```text
Sticky Event（粘性事件）
数据倒灌
```

---

## 2. 只能主线程 setValue

错误：

```java
liveData.setValue(data);
```

如果在子线程：

```java
java.lang.IllegalStateException
```

只能：

```java
liveData.postValue(data);
```

限制较大。

---

## 3. postValue 会丢数据

源码：

```java
mPendingData
```

只有一个缓存。

例如：

```java
postValue(1);
postValue(2);
postValue(3);
```

主线程还没处理时：

```text
1 丢失
2 丢失
3 保留
```

最终：

```text
只收到 3
```

源码核心：

```java
mPendingData = value;
```

不断覆盖。

---

## 4. 不支持背压（BackPressure）

例如：

```java
蓝牙数据
Socket数据
CAN总线数据
```

每秒：

```text
1000条
```

LiveData：

```text
全部切主线程
主线程处理不过来
```

容易：

```text
卡顿
ANR
```

---

## 5. Transformations 性能差

例如：

```java
Transformations.map()
```

```java
Transformations.switchMap()
```

链式复杂：

```text
LiveData
 ↓
map
 ↓
switchMap
 ↓
MediatorLiveData
 ↓
UI
```

维护困难。

---

## 6. 不支持流式操作

RxJava：

```java
map
flatMap
zip
merge
debounce
```

Flow：

```kotlin
map
flatMapLatest
combine
filter
```

LiveData：

```text
几乎没有
```

扩展能力弱。

---

## 7. 生命周期耦合严重

observe：

```java
observe(this, observer);
```

必须：

```text
LifecycleOwner
```

导致：

```text
Repository层
Domain层
SDK层
```

使用困难。

---

## 8. 不适合一次性事件

例如：

```text
Toast
Dialog
Navigation
支付成功
登录成功
```

LiveData：

```text
天然粘性
```

会重复消费。

# 二、源码层面原因

核心类：

```java
LiveData.java
```

内部：

```java
SafeIterableMap
```

保存观察者：

```java
ObserverWrapper
```

数据版本：

```java
mVersion
```

---

发送：

```java
setValue()
```

↓

```java
mVersion++
```

↓

```java
dispatchingValue()
```

↓

```java
observer.onChanged()
```

---

观察者记录：

```java
mLastVersion
```

判断：

```java
mLastVersion < mVersion
```

成立：

```java
回调
```

因此新注册观察者：

```java
observe()
```

也会收到最新值。

这就是：

```text
Sticky Event
```

根本原因。

# 三、如何解决数据倒灌

## 方案1：SingleLiveEvent

Google曾推荐。

```java
public class SingleLiveEvent<T>
        extends MutableLiveData<T> {

    private final AtomicBoolean pending =
            new AtomicBoolean(false);

    @Override
    public void observe(
            LifecycleOwner owner,
            Observer<? super T> observer) {

        super.observe(owner, t -> {
            if (pending.compareAndSet(true,false)) {
                observer.onChanged(t);
            }
        });
    }

    @Override
    public void setValue(T t) {
        pending.set(true);
        super.setValue(t);
    }
}
```

适合：

```text
Toast
Navigation
```

缺点：

```text
多个Observer有问题
```

---

## 方案2：Event包装

Google官方推荐。

```java
public class Event<T> {

    private T data;

    private boolean handled;

    public Event(T data){
        this.data = data;
    }

    public T getContentIfNotHandled(){
        if(handled){
            return null;
        }
        handled = true;
        return data;
    }
}
```

ViewModel：

```java
MutableLiveData<Event<String>> loginEvent;
```

Activity：

```java
loginEvent.observe(this,event->{

    String msg =
            event.getContentIfNotHandled();

    if(msg != null){
        showToast(msg);
    }
});
```

# 五、Flow 替代方案（推荐）

Google 当前更推荐：

```kotlin
StateFlow
SharedFlow
```

---

## StateFlow 替代 LiveData

ViewModel：

```kotlin
private val _state =
    MutableStateFlow(User())

val state = _state.asStateFlow()
```

更新：

```kotlin
_state.value = user
```

UI：

```kotlin
lifecycleScope.launch {
    repeatOnLifecycle(
        Lifecycle.State.STARTED
    ) {
        viewModel.state.collect {
            render(it)
        }
    }
}
```

优点：

```text
线程安全
支持协程
支持操作符
性能更好
```

---

## SharedFlow 替代 SingleLiveEvent

一次性事件：

```kotlin
private val _event =
    MutableSharedFlow<String>()
```

发送：

```kotlin
_event.emit("登录成功")
```

接收：

```kotlin
collect {
    toast(it)
}
```

特点：

```text
无数据倒灌
支持多观察者
支持缓冲区
```

# 六、面试标准回答（3分钟）

面试官：LiveData有哪些缺点？

可以回答：

> LiveData最大的优点是生命周期感知，但存在几个明显缺陷：
> 
> 1. 数据倒灌（Sticky Event），页面重建后会收到旧数据。
>     
> 2. setValue只能主线程调用，postValue高频更新会丢数据。
>     
> 3. 不支持背压，高频数据场景性能较差。
>     
> 4. 不适合一次性事件，如Toast、导航、支付结果。
>     
> 5. 缺少RxJava和Flow丰富的操作符。
>     
> 6. 生命周期耦合较重，不利于跨层使用。
>     
> 
> 实际项目中通常采用 Event 包装类、SingleLiveEvent 解决事件重复消费问题，而新项目更推荐使用 StateFlow 管理状态、SharedFlow 管理事件，能够彻底解决 LiveData 的大部分缺陷。

### 面试加分点

如果面试官继续追问：

**“为什么 LiveData 会产生数据倒灌？”**

直接回答：

> 因为 LiveData 内部通过 `mVersion` 和 `mLastVersion` 机制判断是否分发数据，新注册 Observer 的 `mLastVersion=-1`，而当前数据版本已经大于 -1，因此会立即回调最近一次数据，从而导致页面重建后再次收到旧数据。
