
## 一、Kotlin 编译原理与字节码

### 1. `suspend` 函数的本质是什么？编译后变成什么？

`suspend` 函数通过 **CPS（Continuation Passing Style）变换** 编译为状态机：

```kotlin
// 我们写的代码
suspend fun fetchUser(): User {
    val token = getToken()       // 挂起点1
    val user = getUser(token)    // 挂起点2
    return user
}

// 编译器变换后（伪代码）
fun fetchUser(continuation: Continuation<User>): Any? {
    val sm = continuation as? FetchUserStateMachine
        ?: FetchUserStateMachine(continuation)

    when (sm.label) {
        0 -> {
            sm.label = 1
            val result = getToken(sm)
            if (result == COROUTINE_SUSPENDED) return COROUTINE_SUSPENDED
        }
        1 -> {
            val token = sm.result as String
            sm.label = 2
            val result = getUser(token, sm)
            if (result == COROUTINE_SUSPENDED) return COROUTINE_SUSPENDED
        }
        2 -> {
            val user = sm.result as User
            sm.continuation.resume(user)
        }
    }
}
```

**核心要点：**
- `Continuation` 封装了"协程剩余要执行的代码"和当前状态
- 每个挂起点对应一个状态机分支
- `COROUTINE_SUSPENDED` 标记协程已挂起 

### 2. 扩展函数编译后是什么？

```kotlin
// Kotlin
fun String.addExclamation(): String = this + "!"

// 编译为 Java 静态方法
public static String addExclamation(String $this) {
    return $this + "!";
}
```

- 编译为 **静态方法**，接收者作为第一个参数
- 不能访问私有成员
- 无运行时开销

### 3. `inline` 函数的原理与限制

```kotlin
inline fun <T> measureTime(block: () -> T): T {
    val start = System.currentTimeMillis()
    val result = block()
    println("耗时: ${System.currentTimeMillis() - start}ms")
    return result
}
```

- 编译器将函数体 **内联** 到调用处，消除 Lambda 创建开销
- 限制：不能递归调用、不能访问私有成员（除 `@PublishedApi`）、不能用于接口方法

### 4. `reified` 泛型是如何实现的？

```kotlin
inline fun <reified T> Gson.fromJson(json: String): T {
    return fromJson(json, T::class.java)
}
```

- 普通泛型在编译后擦除为 `Object`
- `reified` 配合 `inline`，编译器在调用处直接替换为具体类型
- 只能在 `inline` 函数中使用

### 5. `data class` 编译后生成哪些方法？

- `equals()`、`hashCode()`、`toString()`
- `copy()`（浅拷贝）
- `componentN()`（解构声明）
- 不能继承另一个 `data class`

## 二、协程底层源码

### 6. 协程调度器 `Dispatchers.IO` 的线程池实现

```kotlin
// CoroutineScheduler.kt (Kotlin 1.7.20)
internal object DefaultIoScheduler : CoroutineDispatcher() {
    private val default = UnlimitedIoScheduler.limitedParallelism(
        SystemProp(IO_PARALLELISM_PROPERTY_NAME, 64.coerceAtLeast(AVAILABLE_PROCESSORS))
    )
}
```

- `Dispatchers.IO` 默认 **64 线程**（或 CPU 核心数取大值）
- 使用 `CoroutineScheduler` 实现，基于工作窃取算法
- **误区**：在 IO 线程池上执行 CPU 密集型任务会导致线程竞争、缓存失效 

### 7. `withContext` 如何实现线程切换？

```kotlin
suspend fun <T> withContext(context: CoroutineContext, block: suspend CoroutineScope.() -> T): T {
    val oldContext = currentCoroutineContext()
    val newContext = oldContext + context
    return newContext[ContinuationInterceptor]!!.interceptContinuation { cont ->
        // 调度 cont 到新线程
    }.invokeCancellable(block)
}
```

**关键步骤：**
1. 保存当前协程状态到 `Continuation`
2. `CoroutineDispatcher.dispatch` 将 `Continuation` 提交到目标线程
3. 目标线程恢复协程执行 

### 8. 协程异常传播的树形模型

```kotlin
// JobSupport.kt
public open fun childCancelled(cause: Throwable): Boolean {
    if (cause is CancellationException) return true
    return cancelImpl(cause) && handlesException
}
```

**传播流程：**
1. 子协程抛出异常 → 通知父 Job
2. 父 Job 触发 `childCancelled()` → 取消自身
3. 连带取消所有其他子协程
4. 异常继续向上传递至根协程

**解决方案：**
- `SupervisorJob`：重写 `childCancelled()` 返回 `false`，阻止异常传播
- `CoroutineExceptionHandler`：顶层兜底（仅对根协程生效）
- `supervisorScope`：局部隔离舱 

### 9. 为什么 `viewModelScope` 默认使用 `SupervisorJob`？

```kotlin
// ViewModel.kt
public val viewModelScope: CoroutineScope
    get() = CoroutineScope(SupervisorJob() + Dispatchers.Main.immediate)
```

- ViewModel 内任务通常独立（多个 LiveData/Flow 更新）
- 避免一个数据库操作崩溃导致所有后台任务中止

### 10. `Flow` 的冷流实现原理

```kotlin
fun numbersFlow(): Flow<Int> = flow {
    for (i in 1..5) {
        delay(100)
        emit(i)
    }
}
```

- `flow {}` 创建 `SafeFlow` 对象
- `collect` 时才执行 Lambda，每次收集重新执行
- `emit` 调用 `collector.emit(value)` 向下游传递

### 11. `StateFlow` vs `SharedFlow` 源码差异

| 特性 | StateFlow | SharedFlow |
|------|-----------|------------|
| 内部实现 | `StateFlowImpl`（原子引用） | `SharedFlowImpl`（环形缓冲区） |
| 重放 | 始终持有最新值 | 可配置 `replay` |
| 并发安全 | `AtomicReference` | `synchronized` 块 |
| 背压 | 新值覆盖旧值 | `BufferOverflow` 策略 |

### 12. `stateIn` 的 `SharingStarted` 策略

```kotlin
val uiState = flow.stateIn(
    scope = viewModelScope,
    started = SharingStarted.WhileSubscribed(5000),
    initialValue = UiState.Loading
)
```

- `Eagerly`：立即启动，scope 取消时停止
- `Lazily`：第一个订阅者出现时启动
- `WhileSubscribed(5000)`：最后订阅者消失后保持 5 秒（防止配置变化时重启）

## 八、源码阅读题

### 30. 阅读以下协程代码，分析输出

```kotlin
fun main() = runBlocking {
    val job = launch {
        try {
            repeat(1000) { i ->
                println("Job: sleeping $i ...")
                delay(500L)
            }
        } finally {
            println("Job: I'm running finally")
            delay(1000L)
            println("Job: After delay in finally")
        }
    }
    delay(1300L)
    println("Main: I'm tired of waiting!")
    job.cancelAndJoin()
    println("Main: Now I can quit.")
}
```

**输出：**
```
Job: sleeping 0 ...
Job: sleeping 1 ...
Job: sleeping 2 ...
Main: I'm tired of waiting!
Job: I'm running finally
Main: Now I can quit.
```

**"After delay in finally" 不会打印！** 因为协程被取消后，`finally` 块中调用 `delay()` 等挂起函数会立即抛出 `CancellationException`。

**修复：**
```kotlin
finally {
    withContext(NonCancellable) {
        delay(1000L)
        println("Cleanup done")
    }
}
```


### 31. 以下代码有什么问题？

```kotlin
viewModelScope.launch(Dispatchers.IO) {
    val data = parseLargeJson(response)  // CPU 密集型
    updateUI(data)
}
```

**问题：**
- `Dispatchers.IO` 线程池上限 64，CPU 密集型任务导致线程竞争
- 缓存失效、TLB 刷新，性能下降 300%

**修复：**
```kotlin
viewModelScope.launch(Dispatchers.Default) {
    val data = parseLargeJson(response)
    withContext(Dispatchers.Main) { updateUI(data) }
}
```

## 一、协程 CPS 变换与状态机（源码必考）

### 1. 协程的本质：CPS 变换

Kotlin 协程并非 JVM 原生支持的线程模型，而是编译器层面的 **语法糖**。`suspend` 函数通过 **CPS（Continuation Passing Style）变换** 编译为状态机 。

**核心机制：**
- `suspend` 函数编译后变为接收 `Continuation` 参数的普通函数
- 每个挂起点对应状态机的一个状态分支
- `Continuation` 封装了"协程剩余要执行的代码"和当前上下文

### 2. 状态机源码详解

```kotlin
// 我们写的代码
suspend fun fetchUser(): User {
    println("Step 1")
    val token = getToken()       // 挂起点1
    println("Step 2")
    val user = getUser(token)    // 挂起点2
    return user
}
```

**编译器变换后的伪代码：**

```kotlin
fun fetchUser(continuation: Continuation<User>): Any {
    // 复用已有的 Continuation 或创建新的状态机
    val sm = continuation as? CoroutineImpl ?: object : CoroutineImpl(continuation) {
        var result: Any? = null   // 保存挂起函数的返回值
        var label: Int = 0        // 状态标签
        
        override fun invokeSuspend(result: Result<Any?>): Any {
            this.result = result.getOrThrow()
            return fetchUser(this)  // 递归调用自身，进入状态机
        }
    }
    
    when (sm.label) {
        0 -> {
            println("Step 1")
            sm.label = 1
            // 调用挂起函数，传入自身作为 Continuation
            val result = getToken(sm)
            // 如果返回 COROUTINE_SUSPENDED，说明协程已挂起
            if (result == COROUTINE_SUSPENDED) return COROUTINE_SUSPENDED
            // 如果没挂起，直接进入下一个状态
        }
        1 -> {
            val token = sm.result as String  // 从 result 取出上次返回值
            println("Step 2")
            sm.label = 2
            val result = getUser(token, sm)
            if (result == COROUTINE_SUSPENDED) return COROUTINE_SUSPENDED
        }
        2 -> {
            val user = sm.result as User
            return user  // 完成，返回最终结果
        }
    }
}
```

**关键要点：**
- `CoroutineImpl` 是编译器生成的基类，继承自 `Continuation`
- `label` 记录当前执行位置，每次挂起后递增
- `COROUTINE_SUSPENDED` 是挂起标记，函数立即返回，等待恢复 

### 3. Continuation 恢复流程

```kotlin
// kotlinx.coroutines 中的 Continuation 接口
public interface Continuation<in T> {
    public val context: CoroutineContext
    public fun resumeWith(result: Result<T>)
}
```

**恢复过程：**
1. 挂起函数（如 `delay`）内部调用 `suspendCancellableCoroutine` 创建 `CancellableContinuation`
2. 调度器（如 `DefaultScheduler`）在适当时机调用 `cont.resume(Unit)`
3. `resumeWith` 触发状态机的 `invokeSuspend`，根据 `label` 跳转到正确分支继续执行 

### 4. 协程调度器源码：`Dispatchers.IO` 的线程池

```kotlin
// kotlinx.coroutines 内部实现
internal object DefaultIoScheduler : CoroutineDispatcher() {
    private val default = UnlimitedIoScheduler.limitedParallelism(
        SystemProp(IO_PARALLELISM_PROPERTY_NAME, 64.coerceAtLeast(AVAILABLE_PROCESSORS))
    )
}
```

- `Dispatchers.IO` 默认 **64 线程**（或 CPU 核心数取大值）
- 使用 `CoroutineScheduler` 实现，基于 **工作窃取算法**
- **重要误区**：在 IO 线程池上执行 CPU 密集型任务会导致线程竞争、缓存失效，性能下降 300% 

### 5. 异常传播的树形模型

```kotlin
// JobSupport.kt
public open fun childCancelled(cause: Throwable): Boolean {
    if (cause is CancellationException) return true
    return cancelImpl(cause) && handlesException
}
```

**传播流程：**
1. 子协程抛出异常 → 通知父 Job
2. 父 Job 触发 `childCancelled()` → 取消自身
3. 连带取消所有其他子协程
4. 异常继续向上传递至根协程

**`SupervisorJob` 原理：** 重写 `childCancelled()` 返回 `false`，阻止异常向上传播。这就是为什么 `viewModelScope` 默认使用 `SupervisorJob` —— ViewModel 内任务通常独立，避免一个数据库操作崩溃导致所有后台任务中止

### 8. Choreographer 与 VSync 信号

**Choreographer 承上启下的作用：**

```
承上：接收应用层 callback（Input、Animation、Traversal），缓存到 CallbackQueue
启下：FrameDisplayEventReceiver 接收硬件 VSync 信号，触发 doFrame()
```

**源码流程：**

```java
// Choreographer.java
void scheduleTraversals() {
    mChoreographer.postCallback(
        Choreographer.CALLBACK_TRAVERSAL, mTraversalRunnable, null);
}

// 请求 VSync
private void scheduleVsyncLocked() {
    mDisplayEventReceiver.scheduleVsync();  // 调用 nativeScheduleVsync
}

// 接收 VSync 回调
private final class FrameDisplayEventReceiver extends DisplayEventReceiver {
    @Override
    public void onVsync(long timestampNanos, int builtInDisplayId, int frame) {
        Message msg = Message.obtain(mHandler, this);
        msg.setAsynchronous(true);
        mHandler.sendMessageAtTime(msg, timestampNanos / TimeUtils.NANOS_PER_MS);
    }
    
    @Override
    public void run() {
        doFrame(mTimestampNanos, mFrame);  // 执行所有 callback
    }
}

// doFrame 中按优先级执行
void doFrame(long frameTimeNanos, int frame) {
    doCallbacks(Choreographer.CALLBACK_INPUT, frameTimeNanos);      // 输入
    doCallbacks(Choreographer.CALLBACK_ANIMATION, frameTimeNanos);  // 动画
    doCallbacks(Choreographer.CALLBACK_TRAVERSAL, frameTimeNanos);  // 绘制
}
```

**关键设计：**
- 每次订阅只能收到 **一次** VSync 信号，需要下次信号需重新订阅
- 动画实现就是在处理完一帧后立即再次调用 `scheduleVsync`
- 60Hz 屏幕每 16.6ms 一个 VSync 周期，错过即掉帧 

### 9. Handler 与 Looper 源码

```java
// Looper.java
public static void loop() {
    final Looper me = myLooper();
    final MessageQueue queue = me.mQueue;
    
    for (;;) {
        Message msg = queue.next(); // 阻塞等待，底层使用 epoll
        if (msg == null) return;    // 退出消息
        
        msg.target.dispatchMessage(msg); // 分发到 Handler
        msg.recycleUnchecked();
    }
}
```

- `MessageQueue` 使用 **epoll** 实现阻塞（Android 6.0+）
- `Handler` 的 `sendMessage` 将 Message 加入队列并唤醒 Looper
- `IdleHandler`：消息队列空闲时执行，用于延迟初始化

## 五、Kotlin 编译原理

### 18. `inline` 函数编译原理

```kotlin
// Kotlin 源码
inline fun measureTime(block: () -> Unit) {
    val start = System.currentTimeMillis()
    block()
    println("耗时: ${System.currentTimeMillis() - start}ms")
}

measureTime { doSomething() }
```

**反编译后的 Java 代码：**

```java
// 编译器将函数体直接复制到调用处
long start = System.currentTimeMillis();
doSomething();
System.out.println("耗时: " + (System.currentTimeMillis() - start) + "ms");
```

**核心机制：**
- 编译器将函数体 **内联** 到调用处，消除 Lambda 创建开销
- 高阶函数的 Lambda 参数也会被内联
- 风险：代码膨胀；避免在大型函数中使用 

### 19. `reified` 泛型：突破类型擦除

**Java 泛型的"失忆症"：**
- `List<String>` 和 `List<Integer>` 在 JVM 中都是 `List`（Raw Type）
- 运行时无法获取泛型类型参数

**Kotlin 的解决方案：`inline + reified`**

```kotlin
inline fun <reified T> isA(value: Any) = value is T

fun main() {
    println(isA<String>("abc"))  // true
    println(isA<String>(123))    // false
}
```

**反编译后的 Java 代码：**

```java
public static final void main() {
    // 第一次调用：T 替换为 String
    boolean result1 = "abc" instanceof String;
    System.out.println(result1);

    // 第二次调用：T 替换为 Integer
    boolean result2 = Integer.valueOf(123) instanceof Integer;
    System.out.println(result2);
}
```

**原理三步走：标记 → 搬运 → 替换**
1. **标记**：`inline` 标记函数，`reified` 标记类型参数
2. **搬运**：编译器将函数体复制到调用处
3. **替换**：编译器在调用点将 `T` 硬编码替换为具体类型（如 `String.class`）

**关键结论：**
- JVM 根本不知道 `reified` 的存在！它看到的只是普通的、写死的 `String.class`
- 性能极高，**无反射开销**
- 只能在 `inline` 函数中使用
- **不能在 Java 中调用**（Java 无内联特性）

### 20. 扩展函数编译原理

```kotlin
// Kotlin
fun String.addExclamation(): String = this + "!"

// 编译为 Java 静态方法
public static String addExclamation(String $this) {
    return $this + "!";
}
```

- 编译为 **静态方法**，接收者作为第一个参数
- 不能访问私有成员
- 无运行时开销

### 21. `crossinline` 与 `noinline`

```kotlin
// noinline：禁止某个 lambda 内联
inline fun process(
    doNow: () -> Unit,
    noinline doLater: () -> Unit  // 需要保存/传递的 lambda
) {
    doNow()              // 内联
    scheduler.post(doLater)  // 作为值传递，不能内联
}

// crossinline：允许内联但禁止非局部返回
inline fun runCrossinline(crossinline block: () -> Unit) {
    val runnable = Runnable { block() }  // 内联到 Runnable 中
    runnable.run()
}
```

## 六、综合源码面试题

### 22. 以下协程代码输出什么？为什么？

```kotlin
fun main() = runBlocking {
    val job = launch {
        try {
            repeat(1000) { i ->
                println("Job: sleeping $i ...")
                delay(500L)
            }
        } finally {
            println("Job: I'm running finally")
            delay(1000L)  // 挂起函数！
            println("Job: After delay in finally")
        }
    }
    delay(1300L)
    println("Main: I'm tired of waiting!")
    job.cancelAndJoin()
    println("Main: Now I can quit.")
}
```

**输出：**
```
Job: sleeping 0 ...
Job: sleeping 1 ...
Job: sleeping 2 ...
Main: I'm tired of waiting!
Job: I'm running finally
Main: Now I can quit.
```

**"After delay in finally" 不会打印！** 因为协程被取消后，`finally` 块中调用 `delay()` 等挂起函数会立即抛出 `CancellationException`。

**修复方案：**
```kotlin
finally {
    withContext(NonCancellable) {
        delay(1000L)
        println("Cleanup done")
    }
}
```

### 23. 以下代码有什么问题？
