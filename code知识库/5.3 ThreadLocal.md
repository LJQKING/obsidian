# ThreadLocal 原理及源码解析（Android/Java面试高频）

ThreadLocal几乎是大厂Android面试必问内容。

面试官通常不会只问：

> ThreadLocal是什么？

而是会继续追问：

- 为什么线程隔离？
    
- 数据存在哪里？
    
- ThreadLocalMap是什么？
    
- 为什么会内存泄漏？
    
- Handler为什么用ThreadLocal？
    
- Looper为什么用ThreadLocal？
    

---

# 一、ThreadLocal是什么

官方解释：

> 为每个线程提供独立变量副本。

例如：

```java
ThreadLocal<String> local = new ThreadLocal<>();

local.set("Main");

new Thread(() -> {
    local.set("Worker");
    System.out.println(local.get());
}).start();

System.out.println(local.get());
```

输出：

```java
Main
Worker
```

每个线程拿到的值不同。

---

# 二、ThreadLocal整体结构

很多人误认为：

```java
ThreadLocal
    ↓
存储数据
```

实际上：

```text
Thread
 ├── ThreadLocalMap
 │      ├── ThreadLocalA -> valueA
 │      ├── ThreadLocalB -> valueB
 │      └── ThreadLocalC -> valueC
```

真正存储数据的是：

```java
Thread.threadLocals
```

而不是ThreadLocal对象。

```java
public class Thread implements Runnable {
```

可以看到：

```java
每个线程内部

```


# 四、set()源码

## 调用

```java
threadLocal.set("Hello");
```

源码：

```java
public void set(T value) {

    Thread t = Thread.currentThread();

    ThreadLocalMap map = getMap(t);

    if (map != null)
        map.set(this, value);
    else
        createMap(t, value);
}
```

流程：

```text
set()

↓

获取当前线程

↓

获取ThreadLocalMap

↓

存入Map
```

---

## getMap()

```java
ThreadLocalMap getMap(Thread t) {
    return t.threadLocals;
}
```

注意：

ThreadLocalMap在Thread中。

---

# 五、createMap()

首次存储：

```java
void createMap(Thread t, T firstValue) {

    t.threadLocals =
        new ThreadLocalMap(this, firstValue);
}
```

结构：

```text
MainThread
    ↓
ThreadLocalMap
    ↓
(ThreadLocal,value)
```

---

# 六、get()源码

```java
public T get() {

    Thread t = Thread.currentThread();

    ThreadLocalMap map = getMap(t);

    if (map != null) {

        ThreadLocalMap.Entry e =
            map.getEntry(this);

        if (e != null) {

            return (T)e.value;
        }
    }

    return setInitialValue();
}
```

流程：

```text
获取当前线程

↓

获取ThreadLocalMap

↓

根据ThreadLocal查找Entry

↓

返回value
```

---

# 七、ThreadLocalMap结构

源码：

```java
static class ThreadLocalMap {

    static class Entry
            extends WeakReference<ThreadLocal<?>> {

        Object value;
    }
}
```

重点来了。

Entry：

```java
key = ThreadLocal

value = 用户数据
```

但是：

```java
Entry
   extends WeakReference
```

即：

```text
key：弱引用

value：强引用
```

结构：

```text
Entry

key(ThreadLocal)
    ↓
WeakReference

value(Object)
    ↓
StrongReference
```

---

# 八、为什么使用弱引用

如果是强引用：

```java
ThreadLocal
    ↓
Entry
    ↓
value
```

ThreadLocal被置空：

```java
threadLocal = null;
```

但是：

```java
Entry -> ThreadLocal
```

仍然引用着。

GC无法回收。

---

弱引用：

```java
WeakReference<ThreadLocal<?>>
```

GC后：

```java
key = null
```

自动释放。

# 九、ThreadLocal内存泄漏原理

面试必问。

---

示例：

```java
ThreadLocal<User> local =
        new ThreadLocal<>();

local.set(new User());
```

结构：

```text
Thread
 ↓

ThreadLocalMap
 ↓

Entry

key -> ThreadLocal
value -> User
```

---

后来：

```java
local = null;
```

GC：

```text
ThreadLocal
被回收

key = null
```

变成：

```text
Entry

key = null

value = User
```

此时：

```text
Thread
 ↓
ThreadLocalMap
 ↓
Entry
 ↓
User
```

User还在。

无法回收。

形成：

```text
内存泄漏
```

---

# 十、为什么线程池更容易泄漏

普通线程：

```java
线程结束
```

Thread对象回收。

问题不大。

---

线程池：

```java
ExecutorService
```

线程长期存在：

```text
Thread

↓

ThreadLocalMap

↓

value
```

一直无法释放。

泄漏越来越大。

---

# 十一、remove()源码

解决方案：

```java
threadLocal.remove();
```

源码：

```java
public void remove() {

    ThreadLocalMap m =
        getMap(Thread.currentThread());

    if (m != null)
        m.remove(this);
}
```

删除：

```text
ThreadLocalMap

↓

Entry

↓

null
```

释放value。

# 十二、Android中的应用



# 十四、ThreadLocal面试标准答案

### ThreadLocal实现原理

```text
每个Thread内部维护一个ThreadLocalMap。

ThreadLocal作为Key，
用户数据作为Value。

调用set()时：

Thread.currentThread()
    ↓
ThreadLocalMap
    ↓
存储Entry

调用get()时：

从当前线程的ThreadLocalMap中获取对应Value。

因此实现线程隔离。
```

---

### 为什么会内存泄漏？

```text
ThreadLocalMap.Entry中的Key是弱引用，
Value是强引用。

ThreadLocal被GC后：

key=null

但是Value仍被ThreadLocalMap引用。

如果线程长期存活（线程池），
Value无法释放，形成内存泄漏。

解决方案：

使用完成后调用：

threadLocal.remove();
```
