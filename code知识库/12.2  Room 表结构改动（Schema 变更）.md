**Android Room 表结构改动（Schema 变更）的操作方式和原理**主要围绕**数据库版本递增 + 迁移（Migration）** 来处理。Room 基于 SQLite，变更时必须确保现有用户数据不丢失（生产环境强烈推荐保留数据）。

### 1. 基本操作步骤

1. **修改 Entity 类**（数据模型）：
   - 添加/删除/修改字段（`@ColumnInfo`）、索引、表名等。
   - 示例：新增一个字段
     ```kotlin
     @Entity(tableName = "users")
     data class User(
         @PrimaryKey val id: Int,
         val name: String,
         val age: Int,           // 原有
         val email: String? = null // 新增字段
     )
     ```

2. **修改 @Database 中的版本号**（必须递增，不能跳版本）：
   ```kotlin
   @Database(
       version = 2,  // 从 1 改为 2
       entities = [User::class],
       // autoMigrations = [...] // 可选，Room 2.4+
   )
   abstract class AppDatabase : RoomDatabase() { ... }
   ```

3. **定义迁移路径**（核心）：
   - **推荐优先使用自动迁移（AutoMigration，Room 2.4+）**：适合简单变更。
   - **手动 Migration**：复杂变更必用。

#### 自动迁移（推荐简单场景）
在 `@Database` 中添加注解：
```kotlin
@Database(
    version = 2,
    entities = [User::class],
    autoMigrations = [
        AutoMigration(from = 1, to = 2)
    ]
)
abstract class AppDatabase : RoomDatabase()
```

- **支持的简单变更**：新增/删除表、新增列、部分索引变更等。
- **复杂情况**（删除/重命名表或列）需要 `@AutoMigrationSpec`：
  ```kotlin
  @RenameTable(fromTableName = "old_users", toTableName = "users")
  // 或 @RenameColumn 等
  class MySpec : AutoMigrationSpec
  ```
  然后在 `@AutoMigration` 中指定 `spec = MySpec::class`。

#### 手动 Migration（灵活，推荐生产环境结合使用）
```kotlin
val MIGRATION_1_2 = object : Migration(1, 2) {
    override fun migrate(db: SupportSQLiteDatabase) {
        // 新增列（推荐带默认值，避免旧数据问题）
        db.execSQL("ALTER TABLE users ADD COLUMN email TEXT")
        
        // 或更复杂操作：创建新表、复制数据、删除旧表等
        // db.execSQL("CREATE TABLE ...")
        // db.execSQL("INSERT INTO new_table SELECT ... FROM old_table")
    }
}

// 在构建时添加
Room.databaseBuilder(context, AppDatabase::class.java, "db_name")
    .addMigrations(MIGRATION_1_2 /*, MIGRATION_2_3 ... */)
    .build()
```

**多版本链式迁移**：Room 会自动按顺序执行（如 1→2→3），不要跳版本。

4. **构建数据库时处理缺失迁移**（可选，开发/测试用）：
   ```kotlin
   .fallbackToDestructiveMigration()  // 破坏性迁移：直接删库重建（丢失数据，勿用于生产）
   // 或 .fallbackToDestructiveMigrationFrom(1,2) 指定特定版本
   ```

5. **导出 Schema（强烈推荐）**：
   - 用于测试和自动迁移生成。
   - 在 `build.gradle` 配置 `schemaDirectory`（Room Gradle 插件）或注解处理器选项。
   - 编译后会在指定目录生成 JSON 文件，提交到版本控制。

6. **测试迁移**（使用 `room-testing`）：
   - 添加依赖 `androidx.room:room-testing`。
   - 用 `MigrationTestHelper` 创建旧版本 DB，执行迁移，验证 schema 和数据。

### 2. 原理是什么？

- **Room 内部机制**：
  - Room 使用 `SQLiteOpenHelper` 的 `onUpgrade` / `onCreate`。
  - 启动时，Room 比较**代码中的 `@Database version`** 与**数据库文件 header 中存储的版本**（以及 `room_master_table` 中的 schema hash）。
  - 如果版本不同且没有对应 `Migration`，会抛异常（`IllegalStateException`：schema changed but no migration）。
  - 有 Migration 时，依次执行 `migrate(start, end)` 方法中的 SQL。

- **为什么必须改版本 + 提供 Migration**？
  - 防止 schema 不匹配导致崩溃。
  - 确保数据迁移逻辑明确（ALTER TABLE、数据拷贝等）。
  - Room 会验证迁移后的 schema 是否与 Entity 匹配（通过导出的 JSON）。

- **自动迁移原理**：
  - Room 在编译期对比新旧 schema JSON，自动生成 SQL（类似手动 ALTER/CREATE）。
  - 对于不明确的变更（如重命名），需要 Spec 提供提示。

### 3. 常见场景示例

- **新增列**：`ALTER TABLE xxx ADD COLUMN new_col TEXT DEFAULT 'value'`
- **删除列**：无法直接 DELETE COLUMN（SQLite 限制），需**创建新表 → 拷贝数据 → 删除旧表 → 重命名**（手动 Migration）。
- **表重命名/拆分**：类似上述，结合 AutoMigrationSpec 或手动 SQL。
- **数据转换**：在 `migrate()` 中执行 UPDATE 语句或复杂逻辑。

### 4. 最佳实践 & 注意事项

- **始终递增版本**，不要跳。
- **生产环境避免 `fallbackToDestructiveMigration`**。
- **测试所有迁移路径**（尤其是链式）。
- **导出 schema 并提交 Git**。
- **复杂变更**优先手动 Migration + 数据备份逻辑。
- **默认值**：新列建议加 `DEFAULT`，否则旧行可能为 NULL 导致问题。
- Room 2.4+ 自动迁移大大简化工作，但仍需测试。
