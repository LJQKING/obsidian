# OkHttp 的多路复用（HTTP/2 Multiplexing）

多路复用（Multiplexing）是 OkHttp 在 **HTTP/2** 中最重要的性能优化之一。

简单来说：

> **一个 TCP 连接上同时传输多个 HTTP 请求和响应，而不需要为每个请求创建独立连接。**

---
# 1. HTTP/1.1 存在的问题

假设一个页面需要请求：

```text
user
banner
news
video
comment
```

HTTP/1.1 下：

```text
TCP1 --> user
TCP2 --> banner
TCP3 --> news
TCP4 --> video
TCP5 --> comment
```

或者：

```text
TCP1
 ├── user
 ├── banner
 ├── news
 ├── video
 └── comment
```

但后者会产生：

### Head Of Line Blocking（队头阻塞）

```text
请求1
请求2
请求3
请求4
```

必须：

```text
请求1完成
 ↓
请求2开始
 ↓
请求3开始
```

前面的请求慢：

```text
video 10s
```

后面全部等待。

---

# 2. HTTP/2 解决方案

HTTP/2 引入：

```text
Stream（流）
Frame（帧）
```

一个 TCP：

```text
TCP Connection
```

内部：

```text
Stream1 -> user
Stream3 -> banner
Stream5 -> news
Stream7 -> video
Stream9 -> comment
```

同时传输：

```text
TCP
 ├── Frame(Stream1)
 ├── Frame(Stream3)
 ├── Frame(Stream5)
 ├── Frame(Stream7)
 └── Frame(Stream9)
```

无需等待。

---

# 3. 数据传输过程

例如：

```java
GET /user
GET /banner
GET /news
```

发送后：

```text
TCP
 ├─ Frame(stream=1)
 ├─ Frame(stream=3)
 ├─ Frame(stream=5)
```

服务端返回：

```text
Frame(stream=5)
Frame(stream=1)
Frame(stream=3)
```

顺序无所谓。

因为：

```text
streamId
```

可以标识属于哪个请求。

```text

```

# 10. 多路复用为什么快

## 减少 TCP 建立

传统：

```text
100请求
100 TCP
```

需要：

```text
三次握手
四次挥手
```

---

HTTP/2：

```text
100请求
1 TCP
```

---

## 减少 TLS 握手

HTTPS：

```text
TLS Handshake
```

耗时通常：

```text
几十~几百ms
```

多路复用后：

```text
只握手一次
```

---

## 避免连接竞争

HTTP/1.1：

```text
连接池
最大5个连接
```

第6个请求等待。

---

HTTP/2：

```text
1连接
100 Stream
```

直接发送。


---

### Q4：多路复用是否解决队头阻塞？

**应用层解决了。**

```text
HTTP/2 Stream
```

之间不会互相阻塞。

但是：

```text
TCP层仍存在队头阻塞
```

因为 TCP 丢包后必须重传。

这也是 HTTP/3（QUIC/UDP）出现的原因。

---

# 一句话总结

OkHttp 的多路复用本质上是：

```text
一个 TCP连接
      ↓
一个 Http2Connection
      ↓
多个 Http2Stream
      ↓
多个请求并发传输
```

通过 HTTP/2 的 **Stream + Frame** 机制，让数十甚至上百个请求共享同一个 TCP/TLS 连接，大幅减少握手开销、连接创建开销和请求等待时间，是 OkHttp 高性能的核心能力之一。
