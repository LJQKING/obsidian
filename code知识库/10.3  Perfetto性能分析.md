
## 一、卡顿分析工具链：Systrace vs Perfetto

### 1.1 工具演进与选择

| 特性    | Systrace（旧版）    | Perfetto（Android 10+ 推荐）    |
| ----- | --------------- | --------------------------- |
| 追踪时长  | 最多 5 秒          | 支持小时级连续追踪                   |
| 数据格式  | 文本 HTML         | Protobuf 二进制，节省 50%+ 空间     |
| 数据源   | ftrace + atrace | 新增内存、GPU、Binder 等十余种        |
| UI 体验 | 基础网页            | 现代化 Web UI，支持搜索/对比          |
| 跨平台   | 仅 Android       | 支持 Android / Linux / Chrome |

**结论**：Android 10 以上设备优先使用 Perfetto，Systrace 作为兼容备选。

## 二、Perfetto 实战：捕获与分析卡顿

### 2.1 捕获 Trace 的三种方式

**方式一：命令行（最灵活）**

```bash
# 基础配置：捕获 10 秒，包含图形、视图、CPU 调度
adb shell perfetto \
  -c - --txt \
  -o /data/misc/perfetto-traces/trace \
  <<EOF
buffers: { size_kb: 65536 }
data_sources: {
  config {
    name: "linux.ftrace"
    ftrace_config {
      ftrace_events: "sched/sched_switch"
      ftrace_events: "gfx"
      ftrace_events: "view"
      atrace_categories: "gfx"
      atrace_categories: "view"
      atrace_categories: "wm"
      atrace_categories: "am"
    }
  }
}
duration_ms: 10000
EOF

# 拉取到本地分析
adb pull /data/misc/perfetto-traces/trace ./trace.perfetto-trace
```

**方式二：Android Studio Profiler（可视化）**

1. 打开 **Profiler** → 选择进程 → **CPU** 标签
2. 选择 **System Trace (Perfetto)** → 点击 **Record**
3. 复现卡顿操作 → **Stop** → 自动在浏览器打开分析

**方式三：代码埋点（精准定位业务代码）**

```kotlin
import android.os.Trace

// 在关键路径插入追踪标记
override fun onBindViewHolder(holder: ViewHolder, position: Int) {
    Trace.beginSection("RV.onBindViewHolder")
    // ... 绑定逻辑
    Trace.endSection()
}
```

### 2.2 Perfetto 分析关键指标

打开 [ui.perfetto.dev](https://ui.perfetto.dev) 加载 trace 文件，关注以下区域：

| 检查区域 | 正常指标 | 异常信号 |
|---------|---------|---------|
| **Frame Timeline** (Android 12+) | 每帧 < 16.6ms (60fps) | 红色长条表示掉帧/Jank |
| **UI Thread** | `Choreographer#doFrame` 均匀分布 | 单帧耗时 > 16ms 或出现空白间隙 |
| **RenderThread** | `DrawFrame` 快速完成 | GPU 渲染队列堆积 |
| **Alerts 面板** | 无警告 | 自动标记 "Jank" 或 "Long frame" |

**卡顿定位流程**：

```
发现掉帧 → 点击红色 Frame → 查看对应时间轴的 UI Thread
         → 找到耗时长的 Trace 切片（如 View#measure、RV#onLayout）
         → 展开调用栈定位到具体方法
```

## 三、RecyclerView 滑动卡顿优化

RecyclerView 是列表卡顿的重灾区，优化需从 **布局、绑定、缓存、预加载** 四个维度入手。

### 3.1 布局层级优化（核心）

**反例：深层嵌套导致 measure 耗时**

```xml
<!-- ❌ 避免：3 层 LinearLayout 嵌套，measure 次数指数增长 -->
<LinearLayout orientation="vertical">
    <LinearLayout orientation="horizontal">
        <ImageView />
        <LinearLayout orientation="vertical">
            <TextView />
            <TextView />
        </LinearLayout>
    </LinearLayout>
</LinearLayout>
```

**正例：ConstraintLayout 扁平化**

```xml
<!-- ✅ 推荐：单层 ConstraintLayout 实现相同效果 -->
<androidx.constraintlayout.widget.ConstraintLayout
    android:layout_width="match_parent"
    android:layout_height="wrap_content">

    <ImageView
        android:id="@+id/avatar"
        app:layout_constraintStart_toStartOf="parent"
        app:layout_constraintTop_toTopOf="parent" />

    <TextView
        android:id="@+id/title"
        app:layout_constraintStart_toEndOf="@id/avatar"
        app:layout_constraintTop_toTopOf="parent" />

    <TextView
        android:id="@+id/subtitle"
        app:layout_constraintStart_toEndOf="@id/avatar"
        app:layout_constraintTop_toBottomOf="@id/title" />
</androidx.constraintlayout.widget.ConstraintLayout>
```

**效果**：使用 ConstraintLayout 可将 item 的 measure + layout 耗时从 **~8ms 降至 ~2ms**。

### 3.2 onBindViewHolder 轻量优化

```kotlin
class OptimizedAdapter : ListAdapter<Item, VH>(DiffCallback()) {

    override fun onCreateViewHolder(parent: ViewGroup, viewType: Int): VH {
        // ❌ 避免：在 onCreateViewHolder 中解析复杂数据
        // ✅ 只负责 inflate 布局
        val binding = ItemLayoutBinding.inflate(
            LayoutInflater.from(parent.context), parent, false
        )
        return VH(binding)
    }

    override fun onBindViewHolder(holder: VH, position: Int) {
        // ❌ 避免：图片同步解码、数据库查询、复杂计算
        // ✅ 只做视图属性设置
        holder.bind(getItem(position))
    }

    class VH(private val binding: ItemLayoutBinding) : 
        RecyclerView.ViewHolder(binding.root) {
        
        fun bind(item: Item) {
            binding.title.text = item.title
            // 图片加载交给 Glide/Coil，它们内部已做异步处理
            Glide.with(binding.image)
                .load(item.imageUrl)
                .placeholder(R.drawable.placeholder)
                .into(binding.image)
        }
    }
}
```

### 3.3 缓存与预加载策略

```kotlin
recyclerView.apply {
    // 1. 固定高度时启用，跳过 measure 阶段
    setHasFixedSize(true)
    
    // 2. 调整缓存池大小（默认 2 屏）
    setItemViewCacheSize(20)
    
    // 3. 开启预加载（Android 6.0+）
    layoutManager = LinearLayoutManager(context).apply {
        initialPrefetchItemCount = 4  // 预加载 4 个 item
    }
    
    // 4. 多个 RecyclerView 共享缓存池
    setRecycledViewPool(sharedPool)
}
```

### 3.4 局部刷新（避免整项重绑）

```kotlin
// ❌ 避免：notifyDataSetChanged() 导致所有 item 重绘
// ✅ 使用 DiffUtil 增量更新
val diffResult = DiffUtil.calculateDiff(MyDiffCallback(oldList, newList))
diffResult.dispatchUpdatesTo(this)

// 更优：使用 ListAdapter 自动处理 Diff
class MyAdapter : ListAdapter<Item, VH>(DiffCallback()) {
    // submitList 内部自动计算 Diff 并局部刷新
}

// 局部刷新 Payload（只更新部分视图，避免 onBindViewHolder 全量执行）
override fun onBindViewHolder(holder: VH, position: Int, payloads: List<Any>) {
    if (payloads.isEmpty()) {
        super.onBindViewHolder(holder, position, payloads)
        return
    }
    // 只更新点赞数，不重新加载图片
    for (payload in payloads) {
        if (payload == PAYLOAD_LIKE_COUNT) {
            holder.updateLikeCount(getItem(position).likeCount)
        }
    }
}
```

## 四、布局层级优化：从 3 层到 1 层

### 4.1 层级检测工具

| 工具 | 用途 | 操作路径 |
|------|------|---------|
| **Layout Inspector** | 查看运行时 View 树深度 | Android Studio → Tools → Layout Inspector |
| **Systrace/Perfetto** | 分析 measure/layout 耗时 | 查看 `View#measure` 切片时长 |
| **adb shell dumpsys gfxinfo** | 查看帧耗时统计 | 关注 `Draw` / `Prepare` / `Process` / `Execute` 阶段 |

### 4.2 布局优化技巧

**1. 使用 `<merge>` 减少根视图冗余**

```xml
<!-- item_layout.xml -->
<merge xmlns:android="http://schemas.android.com/apk/res/android">
    <!-- 使用 merge 时，此布局直接嵌入父布局，不生成额外 ViewGroup -->
    <ImageView android:id="@+id/icon" ... />
    <TextView android:id="@+id/title" ... />
</merge>
```

**2. 使用 `<ViewStub>` 延迟加载复杂视图**

```xml
<!-- 初始不可见，按需 inflate -->
<ViewStub
    android:id="@+id/stub_complex"
    android:layout="@layout/complex_layout"
    android:layout_width="match_parent"
    android:layout_height="wrap_content" />

// 需要显示时
stub_complex.inflate()
```

**3. 使用 `AsyncLayoutInflater` 异步加载**

```kotlin
// 在后台线程解析布局，避免阻塞 UI
AsyncLayoutInflater(context).inflate(R.layout.complex_item, parent) { view, _, _ ->
    // 回调在主线程，设置到 RecyclerView
}
```

**4. 避免过度绘制**


## 五、主线程耗时操作治理

### 5.1 主线程禁忌清单

| 操作类型 | 危害 | 解决方案 |
|---------|------|---------|
| **网络请求** | 阻塞 UI，ANR 风险 | Retrofit + Kotlin Coroutines (Dispatchers.IO) |
| **数据库读写** | SQL 执行可能耗时 100ms+ | Room + 协程 / RxJava `Schedulers.io()` |
| **SharedPreferences 同步提交** | `commit()` 同步写磁盘 | 改用 `apply()` 异步提交 |
| **大图片解码** | BitmapFactory 同步解码耗时 | Glide/Coil 异步加载 |
| **JSON/XML 解析** | 大数据量解析阻塞 | 移入 `Dispatchers.Default` |
| **文件读写** | IO 阻塞主线程 | `withContext(Dispatchers.IO)` |

### 5.2 协程实战：异步化改造

```kotlin
class MyViewModel : ViewModel() {
    
    private val _items = MutableStateFlow<List<Item>>(emptyList())
    val items: StateFlow<List<Item>> = _items.asStateFlow()

    fun loadData() {
        viewModelScope.launch {
            // ✅ 网络请求在 IO 线程
            val response = withContext(Dispatchers.IO) {
                apiService.fetchItems()
            }
            // ✅ 解析在 Default 线程（CPU 密集型）
            val parsed = withContext(Dispatchers.Default) {
                jsonParser.parse(response)
            }
            // ✅ 回到主线程更新 UI
            _items.value = parsed
        }
    }
}

// Adapter 中只负责轻量绑定
class MyAdapter : ListAdapter<Item, VH>(DiffCallback()) {
    override fun onBindViewHolder(holder: VH, position: Int) {
        val item = getItem(position)
        // 只做视图赋值，不做任何计算/IO
        holder.bind(item)
    }
}
```

## 七、优化优先级总结

| 优先级 | 优化项 | 预期收益 | 实施难度 |
|--------|--------|---------|---------|
| **P0** | 主线程 IO/网络/数据库操作异步化 | 卡顿下降 90%+ | ⭐ |
| **P0** | RecyclerView 图片同步解码改异步 | 卡顿下降 90%+ | ⭐ |
| **P1** | 布局层级扁平化（ConstraintLayout） | 卡顿下降 60-80% | ⭐⭐ |
| **P1** | 移除过度绘制 | 卡顿下降 40-60% | ⭐⭐ |
| **P2** | RecyclerView 缓存池/预加载调优 | 卡顿下降 20-40% | ⭐ |
| **P2** | DiffUtil / Payload 局部刷新 | 刷新效率提升 | ⭐⭐ |
| **P3** | 内存抖动/GC 优化 | 卡顿下降 10-20% | ⭐⭐⭐ |

**核心原则**：先用 Perfetto 定位真实瓶颈，再针对性优化。避免盲目优化，相信数据而非直觉。
