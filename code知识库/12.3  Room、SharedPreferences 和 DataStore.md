**Room、SharedPreferences 和 DataStore 是 Android 中常见的本地数据存储方式**，它们在**存储大小限制**、**文件位置**、**适用场景**和**使用方式**上有明显区别。以下是详细对比和说明（基于 Android 官方文档和常见实践）。

### 1. **存储位置（Location）**
所有三种方式的数据都保存在应用的**私有内部存储**（`/data/data/<package_name>/`），外部应用无法直接访问（沙箱安全）。

- **SharedPreferences**：
  - 位置：`/data/data/<package_name>/shared_prefs/<文件名>.xml`
  - 以 XML 格式存储键值对。

- **DataStore**（Preferences DataStore 和 Proto DataStore）：
  - 位置：`/data/data/<package_name>/files/datastore/<文件名>.preferences_pb`（或 `.pb`）
  - 使用 Protocol Buffers（Proto）二进制格式存储，更高效且类型安全。

- **Room**（基于 SQLite）：
  - 位置：`/data/data/<package_name>/databases/<数据库名>.db`（以及可能的 WAL/ journal 文件）
  - 标准 SQLite 数据库文件。

**注意**：这些路径在 Android 设备上需要 root 权限或使用 ADB 才能查看。生产环境中不要依赖硬编码路径。

### 2. **存储大小限制（Size Limits）**
- **SharedPreferences**：
  - **无严格官方上限**，但**强烈不推荐**存储大（>几百 KB）或复杂数据。
  - 实际限制受设备内存影响（整个 XML 文件会加载到内存）。曾有报告超过 ~1-1.4 MB 会导致内存异常。
  - 适合**极小**的 key-value 数据。

- **DataStore**：
  - **类似 SharedPreferences**，适合**小到中等**数据（推荐用于简单配置）。
  - 没有严格上限，但设计目标是**小型数据集**。Proto DataStore 通过定义 schema 更高效，但大对象仍建议用 Room。

- **Room**：
  - **主要受设备可用存储空间限制**（内部存储或安装位置决定）。
  - SQLite 本身支持 GB 级别数据，Room 适合**结构化、大型、复杂**数据集（数千条记录以上）。

### 3. **适用场景（When to Use）**
| 方式              | 适用场景                                      | 不适合场景                          | 推荐指数 |
|-------------------|-----------------------------------------------|-------------------------------------|----------|
| **SharedPreferences** | 简单配置、用户偏好（主题、登录 Token、开关等） | 大数据、复杂查询、列表数据         | 遗留项目 |
| **DataStore**     | 简单 key-value 或结构化对象（现代替代 SharedPreferences） | 大型/关系型数据、需要 SQL 查询     | **强烈推荐** |
| **Room**          | 结构化数据、列表、离线缓存、复杂查询（用户列表、消息记录等） | 极简单的设置（会过度工程）         | 大多数复杂 App |

**Google 官方推荐**：
- 小型简单数据 → **优先 DataStore**（取代 SharedPreferences）。
- 大型/结构化/需要关系 → **Room**。
- DataStore 不支持部分更新、引用完整性或复杂查询时，用 Room。

### 4. **具体使用方式**

#### **SharedPreferences**（简单但有缺陷）
```kotlin
// 获取
val prefs = getSharedPreferences("my_prefs", Context.MODE_PRIVATE)

// 写入
prefs.edit().apply {
    putString("key", "value")
    putInt("age", 25)
    apply()  // 异步，推荐；或 commit() 同步
}

// 读取
val name = prefs.getString("key", "default")
```

**缺点**：同步 API 可能阻塞 UI、主线程 I/O、无事务、类型不安全。

#### **DataStore**（推荐现代方式）
**依赖**（build.gradle）：
```groovy
implementation "androidx.datastore:datastore-preferences:1.1.1"  // Preferences
// 或 Proto DataStore 需要 protobuf
```

**Preferences DataStore 示例**：
```kotlin
// 定义
val Context.settingsDataStore: DataStore<Preferences> by preferencesDataStore(name = "settings")

// 写入 (Coroutine / Flow)
viewModelScope.launch {
    context.settingsDataStore.edit { preferences ->
        preferences[intPreferencesKey("age")] = 25
        preferences[stringPreferencesKey("name")] = "Grok"
    }
}

// 读取 (Flow)
context.settingsDataStore.data.map { preferences ->
    preferences[stringPreferencesKey("name")] ?: "default"
}
```

**Proto DataStore**（存储自定义对象，更类型安全）：
1. 在 `src/main/proto/` 定义 `.proto` 文件。
2. 生成 Serializer。
3. 使用类似 API。

DataStore **异步、安全、有 Flow 观察**，适合 Kotlin/Coroutines 项目。

#### **Room**（强大但较复杂）
**依赖**：
```groovy
implementation "androidx.room:room-ktx:2.6.1"
kapt "androidx.room:room-compiler:2.6.1"
```

**步骤**：
1. **Entity**（表）：
   ```kotlin
   @Entity
   data class User(@PrimaryKey val id: Int, val name: String)
   ```

2. **DAO**（接口）：
   ```kotlin
   @Dao
   interface UserDao {
       @Query("SELECT * FROM user")
       fun getAll(): Flow<List<User>>
       
       @Insert
       suspend fun insert(user: User)
   }
   ```

3. **Database**：
   ```kotlin
   @Database(entities = [User::class], version = 1)
   abstract class AppDatabase : RoomDatabase() {
       abstract fun userDao(): UserDao
   }
   ```

4. **使用**（单例 + Repository）：
   ```kotlin
   val db = Room.databaseBuilder(context, AppDatabase::class.java, "app.db").build()
   ```

支持 Migration、事务、LiveData/Flow 等。适合复杂查询和大量数据。

### 总结建议
- **简单设置** → **DataStore**（首选）或 SharedPreferences（遗留）。
- **列表、缓存、结构化数据** → **Room**。
- **迁移**：SharedPreferences → DataStore 有官方迁移工具。
- 敏感数据考虑 **EncryptedSharedPreferences** 或 Room + SQLCipher。
- 大文件/媒体 → 使用内部/外部文件存储 + MediaStore，而非以上方式。
