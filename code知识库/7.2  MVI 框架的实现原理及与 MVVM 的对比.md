**Android MVI 框架的实现原理及与 MVVM 的对比**

### 1. MVI 是什么？
**MVI**（Model-View-Intent）是一种响应式架构模式，强调**单向数据流（Unidirectional Data Flow）** 和 **单一可信数据源（Single Source of Truth）**。

- **Model**：这里主要指 **State**（不可变状态），代表 UI 在某一时刻的完整快照。
- **View**：负责渲染状态（State），并将用户操作转化为 **Intent** 发送出去。
- **Intent**：用户意图/动作（如点击按钮、刷新、下拉加载等），是输入。

MVI 受前端 Cycle.js 等框架影响，在 Android 中常结合 **Kotlin Flow / StateFlow + ViewModel** 实现。

### 2. MVI 的核心实现方式

#### 关键概念
1. **不可变 State**（UI 状态）：
   - 使用 `data class` 定义，所有字段用 `val`，通常用 `sealed class` 或 `sealed interface` 处理不同状态（Loading、Success、Error 等）。
   - 整个 UI 的状态集中在一个对象里（避免 ViewModel 中散落多个 LiveData/StateFlow）。

2. **Intent**（用户意图）：
   - 通常定义为 `sealed class Intent`（如 `LoadData`、`Refresh`、`ItemClick(id)` 等）。

3. **Reducer**（归约器）：
   - 纯函数：`(currentState: State, intent: Intent) -> newState`。
   - 负责根据当前状态和 Intent 计算出新状态。

4. **ViewModel** 的职责：
   - 接收 Intent。
   - 通过 Reducer 或 Coroutine 处理业务逻辑（Repository 调用、网络、数据库等）。
   - 更新 State（通常用 `MutableStateFlow`）。
   - 可选：处理 Side Effect（一次性事件，如 Toast、导航、Snackbar）。

#### 典型代码结构（Kotlin + Jetpack）

```kotlin
// 1. State
data class UiState(
    val isLoading: Boolean = false,
    val data: List<Item> = emptyList(),
    val error: String? = null
)

// 2. Intent
sealed class UiIntent {
    data object LoadData : UiIntent()
    data object Refresh : UiIntent()
    data class ItemClick(val id: Int) : UiIntent()
}

// 3. ViewModel
class MyViewModel(private val repository: Repository) : ViewModel() {
    private val _state = MutableStateFlow(UiState())
    val state: StateFlow<UiState> = _state.asStateFlow()

    private val _effect = Channel<UiEffect>()  // Side Effect
    val effect = _effect.receiveAsFlow()

    fun processIntent(intent: UiIntent) {
        viewModelScope.launch {
            when (intent) {
                is UiIntent.LoadData -> loadData()
                // ... 其他 intent
            }
        }
    }

    private suspend fun loadData() {
        _state.update { it.copy(isLoading = true) }
        try {
            val result = repository.getData()
            _state.update { it.copy(isLoading = false, data = result, error = null) }
        } catch (e: Exception) {
            _state.update { it.copy(isLoading = false, error = e.message) }
        }
    }
}
```

#### View 层（Activity/Fragment 或 Compose）
- 使用 `repeatOnLifecycle` 收集 `state` 并渲染 UI。
- 用户操作调用 `viewModel.processIntent(...)`。

**Compose 中更优雅**：直接在 Composable 中 `collectAsState()`。

### 3. MVI 与 MVVM 的对比区别

| 维度              | **MVVM**                                      | **MVI**                                              |
|-------------------|-----------------------------------------------|-----------------------------------------------------|
| **数据流**        | 双向数据绑定（View ↔ ViewModel）              | **单向数据流**（Intent → ViewModel → State → View） |
| **状态管理**      | ViewModel 中多个可变属性 / 多个 LiveData/StateFlow | **单一不可变 State**（Single Source of Truth）      |
| **可预测性**      | 中等（状态可能分散，容易出现不一致）          | **高**（状态是函数结果，纯函数 Reducer）            |
| **测试性**        | 好                                            | **更好**（State 和 Reducer 可轻松单元测试）         |
| **UI 一致性**     | 依赖开发者手动同步                            | 天然保证（UI 永远是 State 的映射）                  |
| **复杂度**        | 适合中小型项目，上手快                        | 代码稍多，适合中大型、复杂交互项目                  |
| **Side Effect**   | 容易分散                                      | 通常单独 Channel/ Flow 处理（避免状态污染）         |
| **与 Jetpack 结合** | 天然（ViewModel + LiveData/Flow + DataBinding） | 优秀（Flow + StateFlow + Compose）                  |

**核心区别总结**：
- **MVVM** 更灵活、代码量少，但状态容易碎片化（多个 `MutableLiveData`），在复杂页面中可能出现“状态爆炸”。
- **MVI** 通过**单向流 + 不可变状态** 强制约束，让状态变化更可追溯、可预测，尤其适合需要严格状态管理的场景（如电商订单、表单多步骤、实时协作）。

MVI 本质上是 **MVVM 的进化版**，在 MVVM 基础上增加了更强的约束（单向 + 单一状态）。

### 4. 优缺点 & 适用场景

**MVI 优点**：
- 状态一致性强，bug 更少。
- 易于时间旅行调试（Debug 时可回放 Intent）。
- Reducer 逻辑清晰，可复用。

**MVI 缺点**：
- 样板代码稍多（State、Intent、Reducer）。
- 学习曲线比 MVVM 陡峭一点。

**推荐使用 MVI 的场景**：
- 复杂业务页面（多状态切换、大量用户交互）。
- 团队协作项目（状态定义清晰，减少沟通成本）。
- 结合 Jetpack Compose 使用效果最佳。

**MVVM 依然是主流**，很多项目混合使用（简单页面 MVVM，核心模块 MVI）。
