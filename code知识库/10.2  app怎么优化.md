
## **一、启动速度优化原理**

### **问题：App 启动流程中哪些环节可以优化？**

#### **启动的四个阶段**

```
User tap icon
    ↓
[1] Process 创建 (fork)
    ↓
[2] Application.onCreate()  ← 关键优化点
    ↓
[3] Activity.onCreate() → onStart() → onResume()  ← UI 优化
    ↓
[4] 首帧显示
```

---

#### **[1] Process 创建阶段 - Zygote fork**

**原理**：

```
Zygote (Java VM)
    ↓
fork() [系统调用] → copy-on-write
    ↓
app process (with preloaded classes)
```

**为什么 fork 快？**

- Zygote 预加载了 ~3000 个系统类（framework 核心类）
- `copy-on-write`：fork 时物理内存共享，写时拷贝
- 避免重复解析这些类的字节码

**优化策略**：

- ❌ 不能加快 fork（系统级别）
- ✅ 减少 app 启动时加载的额外类

---

#### **[2] Application.onCreate() - 关键优化点**

**系统调用栈**：

```java
// ActivityThread.java
private void handleBindApplication(AppBindData data) {
    // 1. 创建 Application 实例
    Application app = data.info.makeApplication(false, mInstrumentation);
    
    // 2. 执行 Application.onCreate()
    mInstrumentation.callApplicationOnCreate(app);
}
```

**这一阶段通常做什么？**

```kotlin
class MyApp : Application() {
    override fun onCreate() {
        // [热点] 各种 SDK 初始化
        initCrashHandler()      // Crash 捕获
        initApm()               // 性能监控
        initAnalytics()         // 数据分析
        initDatabase()          // 数据库创建
        initImageLoader()       // 图片库初始化
        initNetwork()           // 网络库初始化
    }
}
```

**为什么这里是瓶颈？**

- 主线程阻塞
- `StrictMode` 检测时会闪现 ANR 警告
- onCreate() 结束后，系统才能绘制首帧

**原理级优化方案**：

|优化方式|原理|权衡|
|---|---|---|
|**延迟初始化**|将非关键 SDK 从 onCreate 移到后台线程或 Activity.onCreate 后|可能影响功能可用性|
|**懒加载**|使用 `lazy delegate` 或 `lateinit`，第一次使用时初始化|首次使用会有延迟|
|**Provider 优化**|避免 ContentProvider 的重初始化|需要了解 init 流程|
|**类加载器预热**|主线程空闲时预加载热点类|内存开销|

**关键：ContentProvider 自动初始化**

```xml
<!-- AndroidManifest.xml -->
<provider
    android:name="com.google.firebase.FirebaseInitProvider"
    android:authorities="com.yourapp.firebase-init"
    android:initOrder="100"
    android:exported="false" />
```

虽然没在代码中显式调用，但系统会在 Application 创建前自动初始化所有 Provider！

**解决方案**：

```xml
<!-- 延迟初始化 -->
<provider
    android:name="com.google.firebase.FirebaseInitProvider"
    android:authorities="com.yourapp.firebase-init"
    android:initOrder="100"
    android:exported="false"
    tools:node="remove" />  <!-- 手动删除自动初始化 -->

<!-- 在需要时显式初始化 -->
<application>
    <provider
        android:name=".MyInitProvider"
        tools:node="merge" />
</application>
```

#### **[3] Activity.onCreate() - UI 线程优化**

**系统调用流程**：

```java
// ActivityThread.java
private void performLaunchActivity(ActivityClientRecord r, ...) {
    // 1. 创建 Activity 实例
    Activity activity = mInstrumentation.newActivity(...);
    
    // 2. attach() → 初始化 mDecor 等
    activity.attach(appContext, ...);
    
    // 3. onCreate(savedInstanceState)
    mInstrumentation.callActivityOnCreate(activity, r.savedInstanceState);
    
    // 4. onStart()
    // 5. onResume()
}

// 在 onResume 后的下一帧，才执行 ViewRootImpl.performTraversal()
```

**关键问题**：onCreate 中加载大数据会阻塞 UI 线程

```kotlin
class MainActivity : AppCompatActivity() {
    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        setContentView(R.layout.activity_main)
        
        // ❌ 重操作在主线程
        loadLargeDataset()  // 占用主线程 200ms
        // 结果：首帧延迟 200ms
    }
}
```

**优化原理**：

```kotlin
class MainActivity : AppCompatActivity() {
    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        setContentView(R.layout.activity_main)
        
        // ✅ 立即返回，让 UI 绘制
        lifecycleScope.launch(Dispatchers.Default) {
            val data = loadLargeDataset()
            
            // 回到主线程更新 UI
            withContext(Dispatchers.Main) {
                updateUI(data)
            }
        }
    }
}
```

为什么这样快？

- setContentView() 只是解析 XML → 构建 View 树
- onResume() 结束后，系统会调用 `ViewRootImpl.performTraversal()`
- performTraversal → measure → layout → draw（这才是耗时的）
- 如果数据加载与 layout/draw 平行，就不会额外延迟首帧

---

#### **[4] 首帧绘制 - Vsync 同步**

**原理图**：

```
t0        t16.67ms      t33.34ms
|-----------|-----------|
Frame 0    Frame 1    Frame 2

应用绘制
  ↓
GPU 渲染
  ↓
Vsync 信号同步 (60fps = 每 16.67ms 一帧)
  ↓
屏幕显示
```

如果 App 绘制 > 16.67ms，会错过该 Vsync，导致掉帧。

---

### **启动优化的度量标准**

```
冷启动时间 = 用户点击 → 首帧显示的总时间

类别：
- 冷启动：App 进程未启动过
- 温启动：进程存活，Activity 重建
- 热启动：Activity 栈顶，直接显示
```

## **二、页面流畅度（卡帧）原理**

### **问题：为什么滑动列表时卡顿？**

#### **从操作系统到应用的调用链**

```
硬件（触屏）
  ↓
Linux 内核 (input subsystem)
  ↓
Android InputDispatcher
  ↓
ViewRootImpl.InputStage.processPointerEvent()
  ↓
View.dispatchTouchEvent()
  ↓
RecyclerView.onTouchEvent() → scroller.fling()
  ↓
Choreographer 注册下一帧回调
  ↓
[等待 Vsync 信号]
  ↓
ViewRootImpl.doTraversal() 
  → RecyclerView.onLayout()
  → 创建 ViewHolder
  → bindViewHolder() ❌ [热点 - 耗时操作]
  ↓
measure → layout → draw
  ↓
GPU 渲染（OpenGL 命令提交）
  ↓
显示器刷新
```

#### **卡顿的核心原因：主线程耗时**

```kotlin
// ❌ RecyclerView Adapter - 导致卡顿
class MyAdapter : RecyclerView.Adapter<ViewHolder>() {
    override fun onBindViewHolder(holder: ViewHolder, position: Int) {
        // 这个方法在主线程执行，必须在 16.67ms 内完成
        
        val data = dataList[position]
        
        // 耗时操作 1：数据库查询
        val userInfo = database.queryUser(data.userId)  // 50ms ❌
        
        // 耗时操作 2：图片处理
        val bitmap = BitmapFactory.decodeFile(data.imagePath)  // 100ms ❌
        
        // 耗时操作 3：JSON 解析
        val json = JSONObject(data.jsonString)  // 20ms ❌
        
        holder.textView.text = userInfo.name
        holder.imageView.setImageBitmap(bitmap)
    }
}
```

**结果**：

```
Frame 0: 50ms（数据库） + 100ms（图片） + 20ms（JSON）
        = 170ms >> 16.67ms
        → 错过 Vsync，屏幕冻结
        
预计掉帧数 = 170 / 16.67 ≈ 10 帧
```

#### **卡顿检测原理 - 帧率监控**

```kotlin
// 方法 1：Choreographer 回调
class FrameMonitor {
    private var lastFrameTime = 0L
    
    fun startMonitoring() {
        Choreographer.getInstance().postFrameCallback(object : Choreographer.FrameCallback {
            override fun doFrame(frameTimeNanos: Long) {
                val currentTime = System.currentTimeMillis()
                val delta = currentTime - lastFrameTime
                
                if (delta > 16.67) {
                    val droppedFrames = (delta / 16.67).toInt()
                    Log.e("FPS", "掉帧数: $droppedFrames")
                }
                
                lastFrameTime = currentTime
                Choreographer.getInstance().postFrameCallback(this)
            }
        })
    }
}
```

**原理**：

- Choreographer 每收到一次 Vsync 信号就调用注册的回调
- 如果两次回调间隔 > 16.67ms，说明掉帧了
- 掉帧数 = delta / 16.67

#### **优化策略**

|策略|原理|代码示例|
|---|---|---|
|**预加载**|在滑动前加载数据|`DiffUtil.calculateDiff(onWorkerThread)`|
|**懒加载**|不可见的 item 暂不加载|Glide + `skipMemoryCache(true)`|
|**缓存**|避免重复计算|LiveData + MutableStateFlow|
|**后台线程**|数据库查询、JSON 解析移到后台|`Flow { emit(queryDB()) }`|
|**减少 View 层级**|减少 measure/layout 次数|merge layout、ViewStub|

**最佳实践**：

```kotlin
class OptimizedAdapter : RecyclerView.Adapter<ViewHolder>() {
    override fun onBindViewHolder(holder: ViewHolder, position: Int) {
        val data = dataList[position]
        
        // 主线程只做 UI 更新
        holder.textView.text = data.name  // 已预处理
        
        // 异步加载图片（Glide 自动处理线程）
        Glide.with(holder.itemView)
            .load(data.imagePath)
            .into(holder.imageView)
    }
}
```

## **三、稳定性（OOM/ANR）原理**

### **问题：内存泄露导致 OOM 的原理是什么？**

#### **Java 内存模型回顾**

```
Heap（堆）
├─ Young Generation
│  ├─ Eden (100%)
│  ├─ Survivor0 (0%)
│  └─ Survivor1 (0%)
│
├─ Old Generation
│  └─ [对象晋升]
│
└─ Permanent Generation / Metaspace
   └─ 类定义、方法表等
```

#### **GC Roots 与可达性分析**

**GC 判断对象是否活跃的方式**：

```
GC Roots:
  ├─ Stack 中的局部变量引用
  ├─ Static 字段引用
  ├─ JNI 引用
  └─ Monitor 持有的对象

GC 算法（可达性分析）:
  从 GC Roots 出发 → 遍历所有可达对象 → 不可达的对象标记为垃圾
```

#### **内存泄露的典型场景**

**场景 1：Activity 泄露 - 静态引用**

```kotlin
// ❌ 内存泄露
class MyActivity : AppCompatActivity() {
    companion object {
        var instance: MyActivity? = null  // 静态引用
    }
    
    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        instance = this  // ← Activity 被 static 字段持有
    }
}

// Activity 销毁后，instance 仍然是 GC Root
// Activity 的所有成员变量、View 树都无法被回收
```

**泄露链**：

```
GC Roots
  ↓
MyActivity.instance (static)
  ↓
MyActivity 对象
  ↓
DecorView (Activity 的根 View)
  ↓
所有子 View、LayoutParams、ListenerCallback
  ↓
无法回收 → 内存持续增长
```

**检测原理**：

```kotlin
// 使用 WeakReference
companion object {
    var instanceRef = WeakReference<MyActivity>(null)
}

// 或者用 Application Context
// 不要持有 Activity/View 的强引用
```

---

**场景 2：Listener 泄露 - 事件订阅**

```kotlin
// ❌ 内存泄露
class EventBus {
    companion object {
        val listeners = mutableListOf<EventListener>()  // 全局列表
    }
    
    fun subscribe(listener: EventListener) {
        listeners.add(listener)  // ← 永久持有
    }
}

class MyFragment : Fragment() {
    override fun onViewCreated(view: View, savedInstanceState: Bundle?) {
        super.onViewCreated(view, savedInstanceState)
        EventBus.subscribe(object : EventListener {
            override fun onEvent(event: Event) {
                // 做点什么
            }
        })
        // ❌ Fragment 销毁时没有 unsubscribe()
        // EventBus 永远持有这个 listener
        // listener 引用 MyFragment（通过 this）
        // → Fragment 无法回收
    }
}
```

**修复**：

```kotlin
class MyFragment : Fragment() {
    private val listener = EventListener { event ->
        // 处理事件
    }
    
    override fun onViewCreated(view: View, savedInstanceState: Bundle?) {
        super.onViewCreated(view, savedInstanceState)
        EventBus.subscribe(listener)
    }
    
    override fun onDestroyView() {
        super.onDestroyView()
        EventBus.unsubscribe(listener)  // 手动清理
    }
}
```

**场景 3：Handler 泄露 - 延迟任务**

```kotlin
// ❌ 内存泄露
class MyActivity : AppCompatActivity() {
    private val handler = Handler(Looper.getMainLooper())
    
    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        
        // 发送 10 秒后执行的任务
        handler.postDelayed({
            // 此时 Activity 可能已销毁
            Toast.makeText(this, "延迟任务", Toast.LENGTH_SHORT).show()
        }, 10000)
        
        // Activity 销毁时，Handler 的消息队列中仍然有这个任务
        // 任务持有 Activity 引用（闭包捕获 this）
        // → Activity 无法回收
    }
}
```

**泄露链**：

```
MessageQueue (全局)
  ↓
Message (延迟 10s)
  ↓
Callback (闭包)
  ↓
MyActivity (captured this)
  ↓
无法回收
```

**修复**：

```kotlin
class MyActivity : AppCompatActivity() {
    private val handler = Handler(Looper.getMainLooper())
    
    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        
        handler.postDelayed({
            // ...
        }, 10000)
    }
    
    override fun onDestroy() {
        super.onDestroy()
        handler.removeCallbacksAndMessages(null)  // 清理所有任务
    }
}
```

#### **ANR 原理 - 应用无响应**

**定义**：应用的主线程被阻塞超过特定时间

|场景|超时时间|
|---|---|
|用户输入（触摸、按键）|5 秒|
|Service 的 onCreate/onStart/onDestroy|20 秒|
|BroadcastReceiver.onReceive()|10 秒|

**ANR 的触发机制**：

```java
// InputDispatcher.cpp (系统进程)
void InputDispatcher::notifyMotion(...) {
    // 1. 发送触摸事件到 App
    // 2. 等待 App 响应
    
    if (elapsed > 5000ms && !response_received) {
        // 5 秒后仍无响应
        adb_cmd("dumpsys meminfo --unreachable");
        adb_cmd("kill -3 pid");  // 发送 SIGQUIT，生成 stack trace
        
        // 弹出 ANR 对话框
        showAnrDialog("App 无响应");
    }
}
```

**常见 ANR 原因**：

```kotlin
// ❌ ANR 1：主线程 IO 操作
class MyActivity : AppCompatActivity() {
    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        
        val file = File("/sdcard/large_file.bin")
        val bytes = file.readBytes()  // ← 阻塞主线程
        
        // 如果文件很大（几百 MB），可能 > 5s
    }
}

// ❌ ANR 2：主线程数据库查询
class MyActivity : AppCompatActivity() {
    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        
        val users = database.queryAllUsers()  // ← 阻塞主线程
    }
}

// ❌ ANR 3：主线程 Lock 竞争
class MyActivity : AppCompatActivity() {
    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        
        synchronized(singletonLock) {  // ← 等待锁
            initSingleton()
        }
        // 如果另一个后台线程持有 lock，就会阻塞
    }
}
```

**ANR 分析**：

当发生 ANR 时，系统会生成 `traces.txt`：

```
----- pid 1234 at 2024-01-15 10:30:45 -----
Cmd line: com.example.app

...
"main" prio=5 tid=1 BLOCKED
  | group="main" sCount=1 dsCount=0 obj=0x... tid=0x... state=BLOCKED
  - blocked on a java.lang.Object@... (a java.lang.Object)
  - held by tid=3 (UserEventDispatcher)
  at com.example.DatabaseHelper.queryAllUsers (DatabaseHelper.java:123)
  at com.example.MyActivity.onCreate (MyActivity.kt:45)
  ...
```

**修复**：

```kotlin
class MyActivity : AppCompatActivity() {
    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        setContentView(R.layout.activity_main)
        
        // ✅ 在后台线程加载
        lifecycleScope.launch(Dispatchers.IO) {
            val users = database.queryAllUsers()
            
            withContext(Dispatchers.Main) {
                updateUI(users)
            }
        }
    }
}
```

## **四、包体积优化原理**

### **问题：APK 由哪些部分组成？**

```
APK 结构:
├─ AndroidManifest.xml (二进制格式)
├─ classes.dex
│  └─ Java/Kotlin 字节码 (主要占用 20-40%)
├─ lib/
│  ├─ arm64-v8a/
│  │  └─ *.so (native 库，~40-50%)
│  ├─ armeabi-v7a/
│  └─ x86/
├─ res/
│  ├─ drawable/
│  │  └─ 图片资源 (~20-30%)
│  ├─ layout/
│  └─ values/
└─ assets/
   └─ 其他资源文件

总大小 = dex + so + res + assets
```

#### **Dex 优化 - 代码瘦身**

**问题**：64K 方法数限制

```
Android 5.0 以前：
  单个 dex 最多 64K 个方法（2^16 - 1）

Android 5.0+ (MultiDex)：
  classes.dex
  classes2.dex
  classes3.dex
  ...
  
但多 dex 会影响启动速度（加载多个 dex）
```

**优化策略**：

```gradle
// build.gradle
android {
    buildTypes {
        release {
            minifyEnabled true  // ← 代码混淆 + 移除未使用代码
            proguardFiles getDefaultProguardFile('proguard-android-optimize.txt'), 'proguard-rules.pro'
        }
    }
}
```

**混淆原理**：

```
未混淆:
  public class MyClass {
      public void someVeryLongMethodName() { ... }
  }
  
混淆后:
  public class a {
      public void b() { ... }
  }
  
方法符号 somVeryLongMethodName (27 字符) → b (1 字符)
→ 减少常量池大小 → dex 变小
```

**检测方法**：

```bash
# 查看 dex 中的方法数
android-dex --analyze classes.dex

# 或使用 apktool
apktool d app-release.apk
# 查看 apktool.yml 中的 methodRefs 数
```

---

#### **So 库优化 - Native 库瘦身**

**常见情况**：

```
arm64-v8a/
  └─ libopencv.so (50MB)  ← 太大！

总 APK 大小 = 50MB + 资源 = 60-70MB
```

**优化策略**：

1. **按平台分发**

```gradle
splits {
    abi {
        enable true
        reset()
        include 'arm64-v8a'  // 只包含 64 位
        universalApk true     // 保留一个通用版本
    }
}
```

2. **Symbol 移除**

```bash
# 编译时移除调试符号
arm-linux-android-strip libopencv.so
# 减少 50% 体积
```

3. **库的重新编译**

```cmake
# CMakeLists.txt
set(CMAKE_CXX_FLAGS_RELEASE "-Os")  # 优化大小
set(CMAKE_VISIBILITY_INLINES_HIDDEN ON)  # 隐藏符号
```

#### **资源优化 - 图片压缩**

**问题**：高分辨率图片占用大量空间

```
100x100 图片：
  PNG (未压缩): 100 * 100 * 4 byte = 40KB
  JPG (压缩): ~5-10KB
  WebP (更好压缩): ~3-5KB
```

**优化方案**：

```gradle
android {
    buildTypes {
        release {
            // WebP 转换
            bundleRelease {
                // Gradle 7.0+
                resConfigs "webp"
            }
        }
    }
}
```

**原理**：

```
WebP vs PNG:
  - PNG: 无损压缩（LZ77 算法）
  - WebP: 结合有损 + 无损
  → 体积减少 25-35%
```

## **五、耗电（电池）优化原理**

### **问题：哪些操作最耗电？**

#### **电池容量与功率消耗**

```
电池电量 (mAh) = 充电容量
功率消耗 (mW) = 电流 (mA) × 电压 (V)

续航时间 (h) = 电池容量 (mAh) / 平均功率消耗 (mA)

例：3000mAh 电池 + 100mA 平均消耗
  续航 = 3000 / 100 = 30 小时
```

#### **各硬件的功率消耗对比**

|硬件|功率 (mW)|备注|
|---|---|---|
|**CPU**|500-1000|最耗电|
|**屏幕**|300-500|第二耗电|
|**GPS**|100-200|持续定位很耗电|
|**WiFi**|30-50|低功耗|
|**蓝牙**|20-30|低功耗|
|**4G LTE**|50-100|中等|
|**NFC**|5-10|低功耗|

#### **常见耗电问题**

**1. GPS 持续定位**

```kotlin
// ❌ 耗电方案
class LocationTracker {
    private val locationManager = getSystemService(Context.LOCATION_SERVICE) as LocationManager
    
    fun startTracking() {
```

## **六、网络通信底层原理**

### **问题：OkHttp 的连接复用如何实现？**

#### **HTTP 连接池机制**

**TCP 三次握手成本**：

```
没有连接复用：
  Request 1 → TCP 建立 (50ms) → 数据传输 (10ms) → 断开
  Request 2 → TCP 建立 (50ms) → 数据传输 (10ms) → 断开
  Request 3 → TCP 建立 (50ms) → 数据传输 (10ms) → 断开
  总耗时 = 3 × 50ms + 3 × 10ms = 180ms

使用连接池：
  Request 1 → TCP 建立 (50ms) → 数据传输 (10ms) [保持连接]
  Request 2 → 复用连接 (0ms) → 数据传输 (10ms) [保持连接]
  Request 3 → 复用连接 (0ms) → 数据传输 (10ms) [保持连接]
  总耗时 = 50ms + 2 × 0ms + 3 × 10ms = 80ms
  → 性能提升 55%
```

**OkHttp 连接池源码**：

```kotlin
// OkHttp - ConnectionPool.java
class ConnectionPool(
    maxIdleConnections: Int = 5,
    keepAliveDuration: Long = 5,
    timeUnit: TimeUnit = MINUTES
) {
    private val connections = ArrayDeque<RealConnection>()
    private val routeSelection = RouteSelector.Selection()
    
    fun get(route: Route): RealConnection? {
        // 查找可复用的连接
        for (connection in connections) {
            if (connection.isHealthy && connection.route == route) {
                return connection  // ← 复用
            }
        }
        return null
    }
    
    fun put(connection: RealConnection) {
        // 将连接放入池中
        connections.add(connection)
        // 定期清理空闲连接
    }
}
```

**连接复用的条件**：

```
1. 同一个 Host（域名）
2. 同一个 Scheme（HTTP/HTTPS）
3. 同一个 Port
4. 连接仍然活跃（未关闭）
5. 支持 HTTP Keep-Alive
```

#### **HTTP Keep-Alive 机制**

**没有 Keep-Alive**：

```
Request 1:
  GET /api/users HTTP/1.1
  Host: api.example.com
  Connection: close  ← 明确关闭

Response 1:
  HTTP/1.1 200 OK
  Connection: close
  Content-Length: 100
  
  [response body]
  
[TCP 连接关闭]

Request 2: [新建 TCP 连接...]
```

**使用 Keep-Alive**：

```
Request 1:
  GET /api/users HTTP/1.1
  Host: api.example.com
  Connection: keep-alive  ← 保持活跃
  Keep-Alive: timeout=5, max=100  ← 5秒超时，最多100个请求

Response 1:
  HTTP/1.1 200 OK
  Connection: keep-alive
  Keep-Alive: timeout=5, max=99
  Content-Length: 100
  
  [response body]

[TCP 连接保活...]

Request 2: [复用 TCP 连接]
  GET /api/posts HTTP/1.1
```

#### **OkHttp 的拦截器链**

```
┌─────────────────────────────┐
│  用户代码 client.newCall()   │
└──────────────┬──────────────┘
               ↓
┌────────────────────────────┐
```

### **七、并发控制原理 - ConcurrentHashMap**

#### **JDK 1.7 vs JDK 1.8+ 实现**

**JDK 1.7 - 分段锁（Segment）**

```
ConcurrentHashMap (JDK 1.7)
│
├─ Segment[0] (Lock)
│  └─ HashEntry[] → 16 个 bucket
│
├─ Segment[1] (Lock)
│  └─ HashEntry[] → 16 个 bucket
│
├─ Segment[...] (Lock)
│  └─ HashEntry[] → 16 个 bucket
│
└─ Segment[15] (Lock)
   └─ HashEntry[] → 16 个 bucket

默认 16 个 Segment → 最多 16 个线程并发写入
```

**源码**：

```java
// JDK 1.7
public class ConcurrentHashMap<K, V> extends AbstractMap<K, V> {
    final Segment<K, V>[] segments;  // 分段数组
    
    static final class Segment<K, V> extends ReentrantLock {
        HashEntry<K, V>[] table;  // 该段的哈希表
        
        V put(K key, int hash, V value, boolean onlyIfAbsent) {
            lock();  // ← 获取这一段的锁
            try {
                // 只对当前 segment 进行操作
                // 其他线程可以操作其他 segment
            } finally {
                unlock();
            }
        }
    }
}
```

**并发性能**：

```
16 个线程同时操作 ConcurrentHashMap：
  Thread 1 → Segment[hash(key1) % 16] (Lock acquired)
  Thread 2 → Segment[hash(key2) % 16] (Lock acquired, 不同 segment)
  ...
  → 真正并行，不会相互阻塞
```

---

**JDK 1.8+ - Node 级锁**

```
ConcurrentHashMap (JDK 1.8+)
│
└─ Node[] (bucket 数组，初始大小 16)
   ├─ bucket[0] → Node (hash=-1 时用 ConcurrentHashMap$ForwardingNode 标记扩容)
   ├─ bucket[1] → Node → Node (链表)
   ├─ bucket[2] → TreeBin (当链表长度 > 8 时转为红黑树)
   └─ bucket[...] → ...

特点：
- 没有 Segment，直接在 Node 上进行 CAS/synchronized
- 初始化时不创建所有桶，动态扩容
- 锁粒度更细（Node 级而不是 Segment 级）
```

**源码**：

```java
// JDK 1.8+
public class ConcurrentHashMap<K, V> extends AbstractMap<K, V> {
    transient volatile Node<K, V>[] table;
    
    static class Node<K, V> {
        final int hash;
        final K key;
        volatile V val;
        volatile Node<K, V> next;
        
        // CAS 操作原子性
        static final <K, V> boolean casVal(Node<K, V> node, V cmp, V val) {
            return UNSAFE.compareAndSwapObject(node, VALUEOFFSET, cmp, val);
        }
    }
    
    final V putVal(K key, V value, boolean onlyIfAbsent) {
        // 1. CAS 尝试直接设置（无竞争情况）
        if (casVal(node, null, value)) {
            return null;  // ← 无锁完成
        }
        
        // 2. 竞争时，对单个 Node 进行 synchronized
        synchronized (node) {  // ← 锁单个 node，其他 bucket 不受影响
            // 更新 node
        }
    }
}
```

**性能对比**：

```
JDK 1.7:
  最多 16 个线程真正并行

JDK 1.8+:
  理论上数百个线程真正并行（64 个 bucket 时）
  
  实际上，多数情况都不需要锁（CAS 成功）
  只有在哈希碰撞时才使用 synchronized
```

#### **synchronized vs CAS 的选择**

```kotlin
// 低竞争场景：CAS 高效
class LowContention {
    private var counter = 0L
    
    fun increment() {
        // CAS 通常一次就成功
        UNSAFE.compareAndSwapLong(this, COUNTEROFFSET, counter, counter + 1)
        // 成本：~1-5 ns
    }
}

// 高竞争场景：synchronized 更好
class HighContention {
    @Synchronized
    fun increment() {
        counter++
        // synchronized 避免 CAS 自旋浪费 CPU
        // 成本：~100-200 ns (加锁) + 实际操作
    }
}
```

**为什么**：

```
CAS 失败时会自旋（spin）重试：
  for (int i = 0; i < MAX_RETRIES; i++) {
      if (CAS()) return;  // 重试
  }
  // 高竞争时，CPU 空转浪费
  
synchronized 会让线程挂起：
  monitor_enter(obj);  // 竞争不到就挂起，让出 CPU
  ... 实际操作 ...
  monitor_exit(obj);
  // 避免自旋浪费
```
