## 核心组件总览

Handler 机制由四个类协同构成：`Handler`、`Looper`、`MessageQueue`、`Message`，共同实现线程间的消息通信。

## 一、Message 源码分析

`Message` 是数据载体，采用**对象池**设计避免频繁 GC。

```java
// frameworks/base/core/java/android/os/Message.java

public final class Message implements Parcelable {
    public int what;          // 消息标识
    public int arg1, arg2;    // 轻量整型数据
    public Object obj;        // 任意对象
    public long when;         // 触发时间 (uptimeMillis)
    
    // 关键：构成单链表的 next 指针
    /*package*/ Message next;
    
    // 对象池：最多缓存 50 个
    private static Message sPool;
    private static int sPoolSize = 0;
    private static final int MAX_POOL_SIZE = 50;
    
    // 永远用 obtain() 而非 new Message()
    public static Message obtain() {
        synchronized (sPoolSync) {
            if (sPool != null) {
                Message m = sPool;
                sPool = m.next;  // 从池头取出
                m.next = null;
                m.flags = 0;
                sPoolSize--;
                return m;
            }
        }
        return new Message();   // 池空才 new
    }
    
    // 消息处理完后回收到池尾
    void recycleUnchecked() {
        flags = FLAG_IN_USE;
        what = 0; arg1 = 0; arg2 = 0;
        obj = null; when = 0;
        target = null; callback = null; data = null;
        synchronized (sPoolSync) {
            if (sPoolSize < MAX_POOL_SIZE) {
                next = sPool;
                sPool = this;   // 头插法放回池
                sPoolSize++;
            }
        }
    }
}
```

**关键设计点**：`when` 字段决定消息在链表中的排序位置，链表始终按 `when` 升序排列。

## 二、MessageQueue 源码分析

MessageQueue 的核心是**按时间排序的单链表 + Linux epoll 阻塞机制**。

```java
// frameworks/base/core/java/android/os/MessageQueue.java

public final class MessageQueue {
    private final boolean mQuitAllowed;
    private long mPtr; // native 层 NativeMessageQueue 指针
    
    Message mMessages; // 链表头（when 最小）
    
    // ① 入队：按 when 插入链表
    boolean enqueueMessage(Message msg, long when) {
        synchronized (this) {
            msg.when = when;
            Message p = mMessages;
            boolean needWake;
            
            // 插入队头：when=0（立即执行）或早于当前头
            if (p == null || when == 0 || when < p.when) {
                msg.next = p;
                mMessages = msg;
                needWake = mBlocked;  // 如果在阻塞就唤醒
            } else {
                // 按 when 找到插入位置
                needWake = mBlocked && p.target == null && msg.isAsynchronous();
                Message prev;
                for (;;) {
                    prev = p;
                    p = p.next;
                    if (p == null || when < p.when) break;
                }
                msg.next = p;
                prev.next = msg;
            }
            
            if (needWake) nativeWake(mPtr); // 写 eventfd 唤醒 epoll
        }
        return true;
    }
    
    // ② 出队：阻塞等待，返回到期消息
    Message next() {
        long nextPollTimeoutMillis = 0;
        for (;;) {
            // 调用 epoll_wait，超时精确到下一条消息的 when
            nativePollOnce(mPtr, nextPollTimeoutMillis);
            
            synchronized (this) {
                final long now = SystemClock.uptimeMillis();
                Message msg = mMessages;
                
                if (msg != null && msg.target == null) {
                    // Barrier（同步屏障）：跳过同步消息，找异步消息
                    do { msg = msg.next; } 
                    while (msg != null && !msg.isAsynchronous());
                }
                
                if (msg != null) {
                    if (now < msg.when) {
                        // 消息未到期，计算精确等待时长
                        nextPollTimeoutMillis = (int) Math.min(
                            msg.when - now, Integer.MAX_VALUE);
                    } else {
                        // 取出消息
                        mBlocked = false;
                        mMessages = msg.next;
                        msg.next = null;
                        return msg;
                    }
                } else {
                    nextPollTimeoutMillis = -1; // 无消息，永久阻塞
                }
            }
        }
    }
}
```

**epoll 机制要点**：`nativePollOnce` 最终调用 `epoll_wait`，主线程在无消息时**真正睡眠**（不占 CPU），有新消息入队时通过写 `eventfd` 唤醒。

## 三、Looper 源码分析```

```java
// frameworks/base/core/java/android/os/Looper.java

public final class Looper { 
// 每个线程一个 Looper，存在 ThreadLocal 里 
static final ThreadLocal<Looper> sThreadLocal = new ThreadLocal<>();

final MessageQueue mQueue;
final Thread mThread;

// 主线程在 ActivityThread.main() 中已调用
public static void prepareMainLooper() {
    prepare(false); // 主线程不允许退出
}

// 工作线程须手动调用
public static void prepare() {
    prepare(true);
}

private static void prepare(boolean quitAllowed) {
    if (sThreadLocal.get() != null) {
        throw new RuntimeException("Only one Looper per thread");
    }
    sThreadLocal.set(new Looper(quitAllowed));
}

// 核心：无限循环分发消息
public static void loop() {
    final Looper me = myLooper();
    final MessageQueue queue = me.mQueue;
    
    for (;;) {
        Message msg = queue.next(); // 可能阻塞
        if (msg == null) return;    // quit() 触发
        
        final Printer logging = me.mLogging;
        if (logging != null) logging.println(">>>" + msg.target);
        
        // ← 这行是 ANR 检测的核心监控点
        msg.target.dispatchMessage(msg);
        
        if (logging != null) logging.println("<<<" + msg.target);
        
        msg.recycleUnchecked();
    }
}
```

## 四、Handler 源码分析

```java
// frameworks/base/core/java/android/os/Handler.java

public class Handler {
    final Looper mLooper;
    final MessageQueue mQueue;
    final Callback mCallback;
    
    public Handler(Looper looper, Callback callback) {
        mLooper = looper;
        mQueue = looper.mQueue; // 直接持有 MQ 引用
        mCallback = callback;
    }
    
    // ① sendMessage 系列最终都走这里
    private boolean enqueueMessage(MessageQueue queue, 
                                    Message msg, long uptimeMillis) {
        msg.target = this; // 消息记住自己的 Handler
        return queue.enqueueMessage(msg, uptimeMillis);
    }
    
    // ② 消息分发的优先级顺序
    public void dispatchMessage(Message msg) {
        if (msg.callback != null) {
            // 优先级 1：post(Runnable) 附带的 callback
            handleCallback(msg);
        } else {
            if (mCallback != null) {
                // 优先级 2：构造时传入的全局 Callback
                if (mCallback.handleMessage(msg)) return;
            }
            // 优先级 3：子类重写的 handleMessage
            handleMessage(msg);
        }
    }
    
    // postDelayed 底层实现
    public final boolean postDelayed(Runnable r, long delayMillis) {
        return sendMessageDelayed(getPostMessage(r), delayMillis);
    }
    
    public final boolean sendMessageDelayed(Message msg, long delayMillis) {
        if (delayMillis < 0) delayMillis = 0;
        // 延迟时间转换为绝对时间戳
        return sendMessageAtTime(msg, SystemClock.uptimeMillis() + delayMillis);
    }
}
````

## 五、同步屏障（Sync Barrier）

这是 Handler 机制中最精妙也最少人知道的特性，**用于保障 UI 渲染的高优先级**。

```java
// MessageQueue 中插入屏障
public int postSyncBarrier() {
    return postSyncBarrier(SystemClock.uptimeMillis());
}

private int postSyncBarrier(long when) {
    synchronized (this) {
        final int token = mNextBarrierToken++;
        final Message msg = Message.obtain();
        msg.when = when;
        msg.arg1 = token;
        // 关键：target == null，这就是屏障消息的标志
        // 普通消息 target 必须非 null
        
        // 按时间插入链表
        Message p = mMessages;
        if (p == null || when < p.when) {
            msg.next = p;
            mMessages = msg;
        }
        // ...
        return token;
    }
}
```

屏障的工作原理：当 `next()` 遇到 `target==null` 的屏障消息时，会**跳过所有同步消息**，专门搜索并优先执行异步消息（`msg.isAsynchronous()==true`）。

**实际应用**：`ViewRootImpl` 在 `scheduleTraversals()` 中插入屏障 → Choreographer 发送异步 VSYNC 回调消息 → 屏障保障渲染帧优先于普通消息执行 → 渲染完成后 `removeSyncBarrier()` 移除屏障。

## 六、关键问题汇总

|问题|原理|
|---|---|
|主线程 `loop()` 死循环为何不卡死？|`epoll_wait` 让线程真正睡眠，不占 CPU；有消息时被 `eventfd` 唤醒|
|内存泄漏如何产生？|匿名内部类 Handler 持有外部 Activity 引用，延迟消息未执行时 Activity 无法 GC|
|如何避免泄漏？|使用静态内部类 + `WeakReference<Activity>`；`onDestroy` 中调用 `removeCallbacksAndMessages(null)`|
|`post(Runnable)` 和 `sendMessage` 区别？|`Runnable` 被包装进 `msg.callback`，分发时优先级最高|
|ANR 的本质？|主线程 `dispatchMessage` 执行超时（前台 5s / 广播 10s），`InputDispatcher` 超时无响应|
|IdleHandler 是什么？|主线程消息队列空闲时触发的回调，`LeakCanary` 等工具用它在空闲时做检测|

---

整个 Handler 机制的精髓在于：**用 Linux epoll 实现零开销等待，用有序链表实现精确延时，用 ThreadLocal 实现线程隔离，用同步屏障实现优先级调度**。这四点合在一起，构成了 Android UI 系统的底层心跳。
