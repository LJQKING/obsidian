
## 1. **MVC (Model-View-Controller)**

```
┌─────────────┐
│   Model     │
│  (数据逻辑) │
└──────┬──────┘
       │
       ↓
┌─────────────────┐      ┌──────────────┐
│  Controller     │←────→│   View       │
│ (业务逻辑)      │      │ (UI展示)     │
└─────────────────┘      └──────────────┘
```

**特点：**

- **Controller 直接操作 View**，负责更新UI
- **Model 通知 Controller** 数据变化
- View 和 Model 可能直接交互
- Controller 层往往很复杂

**优点：**

- 概念简单，易于理解
- 开发速度快

**缺点：**

- Controller 职责过重，代码量大
- View 和 Model 耦合度高
- 难以单元测试

---

## 2. **MVP (Model-View-Presenter)**

```
┌──────────────┐
│   Model      │
│ (数据逻辑)   │
└────────┬─────┘
         │
         ↓
┌──────────────────┐      ┌──────────────┐
│   Presenter      │←────→│   View       │
│ (中间层逻辑)     │      │ (UI展示)     │
└──────────────────┘      └──────────────┘
```

**特点：**

- **Presenter 是纽带**，完全分离 View 和 Model
- View 只负责展示，不能直接访问 Model
- Presenter 持有 View 的引用，主动更新 View
- 核心是 View 被动接收 Presenter 的指令

**优点：**

- View 和 Model 完全隔离，不会相互耦合
- 逻辑清晰，易于测试
- Presenter 可复用

**缺点：**

- Presenter 仍然持有 View 引用，容易造成内存泄漏
- Presenter 中的业务逻辑仍然很复杂
- 代码量较大

**应用场景：** 早期 Android 开发常用

---

## 3. **MVVM (Model-View-ViewModel)**

```
┌──────────────┐
│   Model      │
│ (数据逻辑)   │
└────────┬─────┘
         │
         ↓
┌──────────────────┐      ┌──────────────┐
│  ViewModel       │      │   View       │
│ (状态和逻辑)     │      │ (UI展示)     │
│  (暴露数据流)    │←─────│ (观察者)     │
└──────────────────┘      └──────────────┘
```

**特点：**

- **ViewModel 暴露数据流**（LiveData、StateFlow等）
- View 通过观察（订阅）ViewModel 的数据变化自动更新
- **没有直接的引用关系**，ViewModel 不持有 View
- 双向绑定（DataBinding 可选）

**优点：**

- 完全解耦，ViewModel 不知道 View 存在
- 自动化的数据更新机制
- 无内存泄漏风险
- 支持数据持久化（Configuration Change 安全）
- 最容易单元测试

**缺点：**

- 学习曲线较陡
- 可能过度设计简单场景
- 需要理解响应式编程（RxJava、Flow）

**应用场景：** 现代 Android 开发推荐方案

---

## 对比表

|维度|MVC|MVP|MVVM|
|---|---|---|---|
|**View与Model的关系**|可直接交互|完全隔离|完全隔离|
|**中间层持有View引用**|是|是|否|
|**数据更新方式**|主动更新|主动更新|被动观察|
|**内存泄漏风险**|低|高|低|
|**可测试性**|差|好|最好|
|**代码复杂度**|低|中|中高|
|**适用场景**|简单界面|中等复杂度|复杂业务逻辑|

## Android 中的实践示例

**MVVM + LiveData 的典型写法：**

```kotlin
// ViewModel
class MyViewModel : ViewModel() {
    private val _userData = MutableLiveData<User>()
    val userData: LiveData<User> = _userData

    fun loadUser(id: String) {
        // 异步加载数据，自动通知观察者
        viewModelScope.launch {
            _userData.value = repository.getUser(id)
        }
    }
}

// Activity/Fragment (View)
class MyActivity : AppCompatActivity() {
    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        val viewModel = ViewModelProvider(this).get(MyViewModel::class.java)
        
        // View 被动观察数据变化
        viewModel.userData.observe(this) { user ->
            updateUI(user)
        }
    }
}
```

## 总结

- **MVC**：适合简单项目，但容易陷入代码混乱
- **MVP**：解决了 MVC 的耦合问题，但存在内存泄漏风险
- **MVVM**：现代推荐方案，尤其配合 LiveData/StateFlow，最符合 Android 设计理念

对于当前的 Android 开发，**优先选择 MVVM + Compose UI** 或 **MVVM + LiveData/StateFlow**。
