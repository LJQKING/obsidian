# ==Android 内存泄露详解==

Android 内存泄露（Memory Leak）是指**对象已经不再使用，但由于仍然被其他对象引用，导致 GC（垃圾回收器）无法回收该对象，占用内存不断增加**。

如果泄露严重，会导致：

- App 卡顿
    
- OOM（OutOfMemoryError）
    
- ANR
    
- 应用崩溃
    

---

# 一、Java GC 为什么回收不了？

Java 使用可达性分析算法（Reachability Analysis）。

当对象与 GC Root 存在引用链时：

```java
GC Root
   ↓
Activity
   ↓
TextView
```

GC 认为对象还在使用。

只有引用链断开：

```java
GC Root

Activity   TextView
```

GC 才会回收。

内存泄露本质：

> 本该断开的引用没有断开。

# 二、Android 常见内存泄露场景

## 1. 静态变量持有 Activity

错误示例：

```java
public class TestManager {

    public static Context mContext;

    public static void init(Context context){
        mContext = context;
    }
}
```

调用：

```java
TestManager.init(this);
```

这里：

```java
this = Activity
```

而：

```java
static变量
```

生命周期：

```java
Application存活
```

导致：

```java
Application
      ↓
TestManager
      ↓
Activity
```

Activity 永远无法释放。

---

### 解决方案

使用：

```java
context.getApplicationContext()
```

```java
mContext = context.getApplicationContext();
```

---

## 2. Handler 内存泄露

经典面试题。

### 错误写法

```java
public class MainActivity extends AppCompatActivity {

    private Handler handler = new Handler(){

        @Override
        public void handleMessage(Message msg){

        }
    };
}
```

实际上：

```java
Handler
   ↓
持有Activity引用
```

如果：

```java
handler.postDelayed(...,60000);
```

Activity退出：

```java
finish();
```

消息仍在：

```java
MessageQueue
    ↓
Message
    ↓
Handler
    ↓
Activity
```

Activity无法回收。

---

### 正确写法

静态内部类：

```java
private static class MyHandler extends Handler {

    private WeakReference<MainActivity> weakReference;

    public MyHandler(MainActivity activity){
        weakReference = new WeakReference<>(activity);
    }

    @Override
    public void handleMessage(Message msg){
        MainActivity activity = weakReference.get();
        if(activity != null){

        }
    }
}
```

---

销毁时：

```java
@Override
protected void onDestroy() {
    super.onDestroy();
    handler.removeCallbacksAndMessages(null);
}
```


## 5. Thread 泄露

### 错误示例

```java
new Thread(){
```

## ==**为什么会发生内存泄露？**==

内存泄露的根本原因是：

> **一个本该被回收的对象，仍然被其他存活对象引用，导致 GC 无法回收。**

Android 使用 JVM 的垃圾回收机制（GC），GC 不是看对象是否使用，而是看对象是否还能被访问到。

例如：

```java
MainActivity activity = new MainActivity();
```

正常情况下：

```text
GC Root
   ↓
MainActivity
```

当 Activity 销毁后：

```java
activity = null;
```

引用链断开：

```text
GC Root

MainActivity
```

GC 就会回收它。

---

### 泄露案例

假设：

```java
public class UserManager {

    private static Context context;

    public static void init(Context ctx){
        context = ctx;
    }
}
```

调用：

```java
UserManager.init(this);
```

此时引用关系：

```text
GC Root
   ↓
ClassLoader
   ↓
UserManager(static)
   ↓
MainActivity
```

即使：

```java
finish();
```

Activity 已经退出：

```text
MainActivity
```

仍被静态变量引用。

GC 会认为：

```text
对象仍然可达
```

因此不会回收。

这就是内存泄露。

# 内存泄露的本质

可以概括成一句话：

```text
生命周期长的对象
持有
生命周期短的对象
```

例如：

|长生命周期对象|短生命周期对象|
|---|---|
|static变量|Activity|
|Singleton|Activity|
|Application|Fragment|
|Thread|Activity|
|Handler消息队列|Activity|

例如：

```text
Application
    ↓
Activity
```

Application 不退出：

```text
Activity 永远无法释放
```

---

# 如何解决？

解决原则：

> **及时断开无用引用。**

---

## 方案1：使用 Application Context

错误：

```java
UserManager.getInstance(this);
```

正确：

```java
UserManager.getInstance(
        getApplicationContext());
```

因为：

```text
Application Context
```

本来就和应用同生命周期。

---

## 方案2：WeakReference（弱引用）

普通引用：

```java
Activity
```

只要存在：

```java
MyManager.activity
```

GC 无法回收。

改成：

```java
WeakReference<Activity>
```

```java
WeakReference<Activity> ref =
        new WeakReference<>(activity);
```

GC 时：

```text
即使 WeakReference 存在
对象仍可被回收
```

示例：

```java
private WeakReference<Activity> activityRef;
```

---

## 方案3：及时注销监听

例如：

```java
registerReceiver(receiver);
```

必须：

```java
unregisterReceiver(receiver);
```

生命周期：

```java
@Override
protected void onDestroy() {
    unregisterReceiver(receiver);
}
```

同理：

```java
EventBus.unregister(this);
```

```java
LiveData.removeObserver(...)
```

```java
RxJava.dispose()
```

---

## 方案4：清理 Handler 消息

错误：

```java
        runnable,
```

Activity 已销毁：

```text

    ↓

    ↓
Handler
    ↓

```

解决：

```java

    handler.removeCallbacksAndMessages(null);

```

## 方案5：关闭线程

错误：

```java
while(true){
}
```

线程一直执行。

解决：

```java
@Override
protected void onDestroy() {
    thread.interrupt();
}
```

或者：

```java
isRunning = false;
```

---

## 方案6：避免非静态内部类

错误：

```java
class MyRunnable implements Runnable {
}
```

编译后实际上：

```java
MyRunnable {
    MainActivity this$0;
}
```

内部自动持有：

```text
MainActivity
```

解决：

```java
static class MyRunnable
        implements Runnable {
}
```

或者：

```java

```

# Android 面试标准答案

面试官：为什么会发生内存泄露？

回答：

> Android 内存泄露本质上是对象已经不再使用，但由于存在引用链导致 GC 无法回收。常见原因是生命周期较长的对象持有生命周期较短的对象，例如静态变量、单例、Handler、线程、监听器持有 Activity 或 Fragment。由于 Android 的 GC 基于可达性分析，只要对象从 GC Root 可达，就不会被回收，因此会产生内存泄露。

面试官：如何解决？

回答：

> 解决思路是及时断开引用链。实际开发中会使用 Application Context 替代 Activity Context，使用 WeakReference 避免强引用，在 onDestroy 中注销 BroadcastReceiver、EventBus、LiveData、RxJava 等监听，同时清理 Handler 消息队列和停止线程。对于排查问题，我通常会使用 LeakCanary 和 Android Profiler 分析 Heap Dump，定位具体的泄露引用链。

这套回答基本符合 Android 中高级工程师面试要求，并且能够延伸到源码层面的 GC Root、引用类型（强引用、弱引用、软引用、虚引用）以及 LeakCanary 原理。
