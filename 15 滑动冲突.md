Android 的“滑动冲突”（Scroll Conflict）本质是：**多个可滚动控件在同一触摸事件链中竞争拦截权**，导致滑动体验异常（卡顿、无法滑动、只响应一方等）。

---

# 一、为什么会发生滑动冲突？

核心原因在于 Android 的事件分发机制（Android事件分发机制）：

### 1. 事件只能被一个 View 消费

一次触摸事件（ACTION_DOWN → MOVE → UP）：

- 只能被一个 View “真正消费”
    
- 父 View 可以决定是否拦截子 View
    

---

### 2. ViewGroup 的拦截机制

关键方法：

```java
onInterceptTouchEvent()
```

流程：

1. 事件先到父 ViewGroup
    
2. 父 View 决定：
    
    - `true` → 拦截（自己处理滑动）
        
    - `false` → 传给子 View
        

---

### 3. 冲突本质

当出现以下结构时容易冲突：

- 外层：RecyclerView / ScrollView
    
- 内层：ViewPager / RecyclerView / ScrollView
    

👉 都想处理滑动手势

结果：

- 外层抢事件
    
- 内层抢事件
    
- 系统不知道该给谁
    

---

# 二、典型滑动冲突场景

## 2. 横向 + 竖向

## 3. 多层嵌套滚动

# 三、解决方案（核心三大类）

---

# 方案1：父容器控制拦截（最基础）

使用：

```java
requestDisallowInterceptTouchEvent(true);
```

## 作用：

告诉父容器：

👉 “别抢我的事件，我要滑”

### 示例（RecyclerView 内滑动）

```java
recyclerView.setOnTouchListener((v, event) -> {
    v.getParent().requestDisallowInterceptTouchEvent(true);
    return false;
});
```

---

# 方案2：自定义 onInterceptTouchEvent（推荐）

核心思路：

👉 根据滑动方向判断谁处理

### 逻辑：

```java
if (absX > absY) {
    // 横向滑动 → 父View处理
    return true;
} else {
    // 竖向滑动 → 子View处理
    return false;
}
```

---

### 示例（经典 ViewPager 冲突解决）

```java
@Override
public boolean onInterceptTouchEvent(MotionEvent ev) {
    switch (ev.getAction()) {
        case MotionEvent.ACTION_DOWN:
            startX = ev.getX();
            startY = ev.getY();
            break;

        case MotionEvent.ACTION_MOVE:
            float dx = ev.getX() - startX;
            float dy = ev.getY() - startY;

            if (Math.abs(dx) > Math.abs(dy)) {
                return true; // 横向拦截
            }
            break;
    }
    return super.onInterceptTouchEvent(ev);
}
```

# 方案3：NestedScrolling 机制（现代推荐）

Android 官方推荐解决方案：

NestedScrolling机制

---

## 作用：

让父子 View **协商滚动，而不是抢事件**

### 支持组件：

- NestedScrollView
    
- RecyclerView
    
- CoordinatorLayout
    
- AppBarLayout
    

---

## 工作流程：

1. 子 View 开始滚动
    
2. 先通知父 View：
    
    ```java
    onStartNestedScroll()
    ```
    
3. 父 View 决定是否参与滚动
    
4. 父子共同消耗滚动距离
    

---

## 优点：

- 不需要手动拦截
    
- 系统级协调
    
- 更顺滑

# 一、滑动冲突的本质（先定结论）

一句话：

> 滑动冲突不是“滑动问题”，而是 **触摸事件在 ViewRootImpl → ViewGroup → View 的分发与拦截链路中，被不同层级抢占导致的路径不一致问题**。

核心链路：

```
ViewRootImpl
   ↓
DecorView
   ↓
ViewGroup.dispatchTouchEvent()
   ↓
onInterceptTouchEvent()
   ↓
childView.dispatchTouchEvent()
   ↓
onTouchEvent()
```

---

# 二、事件从哪里开始？ViewRootImpl

入口类：

ViewRootImpl

当用户手指触摸屏幕时：

## 1. 事件来源（底层）

Linux Input → InputDispatcher → Activity → ViewRootImpl

---

## 2. ViewRootImpl 关键方法

```java
public void handleInputEvent(InputEvent event)
```

最终进入：

```java
deliverPointerEvent()
```

然后分发到：

```java

```

👉 mView = DecorView（顶层 View）

# 三、事件进入 View 树：dispatchTouchEvent 主链路

从 DecorView 开始：

```java
DecorView.dispatchTouchEvent()
```

继续进入：

```java

```

# 四、ViewGroup 是滑动冲突核心

关键类：

ViewGroup

## 1. dispatchTouchEvent 核心逻辑

简化源码流程：

```java
public boolean dispatchTouchEvent(MotionEvent ev) {

    boolean handled = false;

    // Step 1: 是否拦截
    if (onInterceptTouchEvent(ev)) {
        // 自己处理
        handled = onTouchEvent(ev);
    } else {
        // 传给子 View
        handled = child.dispatchTouchEvent(ev);
    }

    return handled;
}
```

# 六、onInterceptTouchEvent 的本质

```java
public boolean onInterceptTouchEvent(MotionEvent ev)
```

作用：

> 决定事件是“自己消费”还是“下发子 View”

---

## 默认行为：

```java
return false;
```

👉 不拦截，交给子 View

---

## 但滑动冲突时：

父 View 会根据 MOVE 判断：

```java
if (horizontalScroll) {
    return true; // 父拦截
}
```

---

# 七、ACTION_DOWN 是“冲突起点”

事件序列：

```
DOWN → MOVE → MOVE → MOVE → UP
```

---

## 关键规则（非常重要）：

### 👉 DOWN 必须决定“归属”

如果父 View 在 DOWN 时：

### 情况1：父拦截

```java
onInterceptTouchEvent(DOWN) = true
```

👉 整个事件序列都给父

---

### 情况2：不拦截

```java
onInterceptTouchEvent(DOWN) = false
```

👉 后续 MOVE 可以再“抢”

---

# 八、滑动冲突的真正发生机制（重点）

## 1. DOWN 阶段

```
事件：DOWN
父：不确定 → 通常不拦截
子：收到 DOWN
```

👉 子 View 获得事件链

---

## 2. MOVE 阶段（冲突爆发）

父开始判断：

```java
if (滑动方向 != 子View方向) {
    return true; // 父抢事件
}
```

---

## 3. 关键机制：事件“中途被截断”

一旦父拦截：

👉 子 View 会收到：

```text
ACTION_CANCEL
```

---

## 这就是滑动冲突最核心现象：

> 子 View 正在滑动 → 突然收到 CANCEL → 滑动失效

---

# 九、ACTION_CANCEL 的源码意义

当父 View 拦截后：

```java
child.dispatchTouchEvent(CANCEL)
```

作用：

> 通知子 View：你这个手势作废了

# 十、完整事件流（滑动冲突本质图）

## 正常情况（无冲突）

```
DOWN → child
MOVE → child
UP   → child
```

## 冲突情况（父抢事件）


# 十二、源码级解决方案对应关系

---

## 1. requestDisallowInterceptTouchEvent

```java
parent.requestDisallowInterceptTouchEvent(true);
```

### 本质：

直接修改 ViewGroup 内部标记：

```java
mGroupFlags |= FLAG_DISALLOW_INTERCEPT;
```

👉 父 View 不再执行 onInterceptTouchEvent

---

## 2. onInterceptTouchEvent（方向判断）

核心逻辑：

```java
dx = moveX - downX;
dy = moveY - downY;

if (abs(dx) > abs(dy)) {
    return true; // 横向
}
```

👉 控制“谁抢事件”

---

## 3. NestedScrolling（现代方案）

NestedScrolling机制

### 本质：

不是“抢事件”，而是：

> 父子 View 协商消耗 scroll distance

流程：

```
child → preScroll → parent
child → consume
parent → remaining scroll
```

# 十三、面试官最爱问的点（重点总结）

👉 多层 ViewGroup 对 touch event 拦截权竞争

## 2. 事件什么时候被抢走？

👉 MOVE 阶段，onInterceptTouchEvent 返回 true

👉 父 View 拦截后通知子 View 取消手势

---

👉 负责把系统输入事件分发到 DecorView

👉 dispatchTouchEvent + onInterceptTouchEvent + 事件序列控制权问题

# 十四、一句话高级总结（面试终结版）

> Android 滑动冲突本质是 ViewRootImpl 将输入事件通过 dispatchTouchEvent 传递到 View 树后，在 ViewGroup 的 onInterceptTouchEvent 阶段，不同层级对 MOVE 事件进行拦截导致事件序列被重新分配，并通过 ACTION_CANCEL 中断子 View 手势链路的过程。
