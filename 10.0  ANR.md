
## 一、ANR的发生机制

### 1. **核心检测原理**

ANR（Application Not Responding）的检测由 `ActivityManagerService` 通过 Watchdog 机制实现：

```
主线程 Handler/Looper 事件循环
  ↓
Message.when 时间检查
  ↓
SystemServer 中的 AMS Watchdog
  ↓
检测主线程是否响应
  ↓
超过阈值 → ANR 触发
```

**具体来说，分为几种 ANR 类型的检测：**

|ANR类型|检测器|超时时间|触发条件|
|---|---|---|---|
|**Input ANR**|InputDispatcher|5s (前台) / 10s (后台)|触摸事件未在时限内处理完|
|**Service ANR**|AMS|20s (前台) / 200s (后台)|Service 生命周期方法阻塞|
|**Broadcast ANR**|AMS|10s (前台) / 60s (后台)|BroadcastReceiver.onReceive 阻塞|
|**ContentProvider ANR**|AMS|10s|Provider 查询操作阻塞|

### 2. **Input ANR 检测源码路径（最常见）**

在 `frameworks/native/services/inputflinger/dispatcher/InputDispatcher.cpp` 中：

```cpp
// 简化的检测逻辑
void InputDispatcher::dispatchOnce() {
    if (currentTime - eventTime > 5000ms) {  // 5秒超时
        ALOGI("Input event not responding for application...");
        // 通知 AMS，触发 ANR
        mPolicy->notifyANR(...)
    }
}
```

**关键流程：**

1. InputDispatcher 分发触摸事件给应用的 ViewRootImpl
2. 事件进入主线程 MessageQueue
3. 如果 5s 内事件未被处理（或处理中一直在工作），InputDispatcher 监测到超时
4. InputDispatcher 调用 `mPolicy->notifyANR()` → 触发 `ActivityManagerService.appNotResponding()`

## 二、为什么会出现 ANR？

### **根本原因：主线程阻塞超过阈值**

```
主线程 Looper.loop()
  ├─ 处理 Input 事件 (onTouchEvent, onClick 等) 
  ├─ 处理 Message 回调 (Handler.post)
  ├─ 处理 View 绘制 (performMeasure/Layout/Draw)
  └─ [如果这里发生 Long-running Task] → ANR
```

### **常见 ANR 场景**

#### 1. **Input ANR - 触摸响应迟缓**

```kotlin
// ❌ 主线程中的同步耗时操作
button.setOnClickListener {
    val data = loadDataFromDb()  // 没有开启子线程
    updateUI(data)
}

// ❌ 在 onTouchEvent 中做耗时工作
override fun onTouchEvent(event: MotionEvent): Boolean {
    calculateComplexMath()  // 主线程阻塞
    return super.onTouchEvent(event)
}
```

#### 2. **Service ANR - Service 生命周期卡死**

```kotlin
// ❌ Service.onCreate() 或 onStartCommand() 中长期阻塞
override fun onStartCommand(intent: Intent?, flags: Int, startId: Int): Int {
    val result = complexOperation()  // 耗时操作，无子线程
    return START_STICKY
}
```

#### 3. **Broadcast ANR - 广播处理超时**

```kotlin
// ❌ BroadcastReceiver.onReceive 中同步网络请求
override fun onReceive(context: Context, intent: Intent) {
    val response = httpClient.newCall(request).execute()  // 同步阻塞
    updateStatus(response)
}
```

#### 4. **主线程消息堆积**

```kotlin
// ❌ Handler 不断 post 大量 Message
for (i in 0..10000) {
    handler.post { heavyWork() }  // MessageQueue 堆积，处理缓慢
}
```

#### 5. **View 绘制耗时**

```kotlin
// ❌ onMeasure/onLayout/onDraw 中做复杂计算
override fun onDraw(canvas: Canvas) {
    for (i in 0..1000) {
        expensiveCalculation()  // 主线程绘制卡顿
    }
    super.onDraw(canvas)
}
```

#### 6. **死锁或其他同步问题**

```kotlin
// ❌ 持有锁的同时等待另一个线程释放锁
synchronized(lock1) {
    while (!condition) {
        Thread.sleep(100)  // 主线程睡眠 + 等待
    }
}
```

## 三、如何调试和解决 ANR

### **1. ANR 日志定位**

ANR 发生时系统会生成以下关键文件：

```bash
# 查看 logcat 中的 ANR 信息
adb logcat | grep ANR

# ANR 日志通常包含：
# - 进程信息和堆栈信息
# - 主线程当前执行的 Method（PID）
# - 各线程的状态
```

**关键日志示例：**

```
E/ActivityManager: ANR in com.example.app (com.example.app/.MainActivity)
E/ActivityManager: PID: 12345
E/ActivityManager: Reason: Input dispatching timed out
E/ActivityManager: Load: 10.05 / 9.87 / 9.45

# 堆栈跟踪 - 找到主线程卡住的位置
Main Stack:
  at java.lang.Thread.sleep(Native Method)
  at com.example.app.MainActivity.heavyWork()
  at com.example.app.MainActivity$1.onClick()
  at android.view.View.performClick()
  at com.example.app.MainActivity.onTouchEvent()
```

### **2. 使用 TraceView 或 Systrace 分析**

```bash
# Systrace 捕获 ANR 时的系统行为
python systrace.py --time 10 -a com.example.app

# 查看主线程时间线：
# - 标记出 LONG_MSG 或阻塞的地方
# - 对比其他进程是否也在阻塞
# - 定位具体耗时函数
```

### **3. 代码层面的解决方案**

#### **方案 A：异步化 - 使用协程/Thread/ExecutorService**

```kotlin
// ❌ 同步版本 - 导致 ANR
button.setOnClickListener {
    val data = loadDataFromDb()  // 主线程阻塞
    updateUI(data)
}

// ✅ 协程版本 - 推荐
button.setOnClickListener {
    lifecycleScope.launch(Dispatchers.Default) {
        val data = loadDataFromDb()  // IO线程
        withContext(Dispatchers.Main) {
            updateUI(data)  // 主线程更新UI
        }
    }
}

// ✅ RxJava 版本
button.setOnClickListener {
    loadDataFromDbObservable()
        .subscribeOn(Schedulers.io())
        .observeOn(AndroidSchedulers.mainThread())
        .subscribe { data -> updateUI(data) }
}

// ✅ Thread + Handler 版本
button.setOnClickListener {
    Thread {
        val data = loadDataFromDb()
        handler.post { updateUI(data) }
    }.start()
}
```

#### **方案 B：优化 Service/BroadcastReceiver**

```kotlin
// ❌ Service.onStartCommand 阻塞
override fun onStartCommand(intent: Intent?, flags: Int, startId: Int): Int {
    val result = complexNetworkCall()  // ANR!
    return START_STICKY
}

// ✅ 使用 IntentService 或 JobService
class MyIntentService : IntentService("MyService") {
    override fun onHandleIntent(intent: Intent?) {
        val result = complexNetworkCall()  // 后台线程
        // 完成后自动停止
    }
}

// ✅ 或者在 Service 中手动开启线程
override fun onStartCommand(intent: Intent?, flags: Int, startId: Int): Int {
    thread {
        val result = complexNetworkCall()
        stopSelf(startId)
    }
    return START_STICKY
}
```

#### **方案 C：优化 BroadcastReceiver**

```kotlin
// ❌ 同步网络请求导致 ANR
override fun onReceive(context: Context, intent: Intent) {
    val response = httpClient.newCall(request).execute()  // ANR!
}

// ✅ 获取 PendingResult，异步处理
override fun onReceive(context: Context, intent: Intent) {
    val pendingResult = goAsync()
    
    lifecycleScope.launch(Dispatchers.IO) {
        try {
            val response = httpClient.newCall(request).execute()
            // 异步处理结果
        } finally {
            pendingResult.finish()  // 必须调用
        }
    }
}
```

#### **方案 D：减少 View 层级和绘制复杂度**

```kotlin
// ❌ onDraw 中频繁创建对象
override fun onDraw(canvas: Canvas) {
    for (i in 0..1000) {
        val paint = Paint()  // 频繁创建
        val rect = RectF()
        canvas.drawRect(rect, paint)
    }
}

// ✅ 预先创建，复用对象
private val paint = Paint().apply { /* 配置 */ }
private val rect = RectF()

override fun onDraw(canvas: Canvas) {
    for (i in 0..1000) {
        rect.set(...)
        canvas.drawRect(rect, paint)  // 复用
    }
}
```

## 四、如何预防 ANR

### **1. 遵循 Android 主线程原则**

```kotlin
// ✅ 16ms 原则（60fps）
// 单帧时间 <= 16ms，这样才不会导致帧率下降
// 主线程应该在 16ms 内完成：
// - onMeasure/onLayout/onDraw
// - 处理一个 Message/Event

// ❌ 错误做法
override fun onDraw(canvas: Canvas) {
    for (i in 0..10000) {
        Thread.sleep(1)  // 总耗时 10s+
    }
}
```

### **2. 使用 ANR Watchdog 监控库**

```kotlin
// 添加依赖
implementation 'com.github.anrwatchdog:anrwatchdog:1.4.0'

// 在 Application 中初始化
class MyApp : Application() {
    override fun onCreate() {
        super.onCreate()
        
        ANRWatchDog()
            .setANRListener { error ->
                // 捕获 ANR，上传到服务器分析
                reportToServer(error)
            }
            .start()
    }
}
```

### **3. 主线程耗时检测工具**

```kotlin
// 自定义 Looper 监控
class MainThreadDetector : Thread() {
    override fun run() {
        Looper.getMainLooper().setMessageLogging { msg ->
            // 记录每个 Message 的处理时间
            val start = System.currentTimeMillis()
            
            // 处理后计时
            val cost = System.currentTimeMillis() - start
            if (cost > 100) {  // 超过 100ms 记录
                Log.w("MainThread", "Slow operation: $cost ms")
            }
        }
    }
}
```

### **4. 代码审查清单**

|检查项|描述|
|---|---|
|**数据库操作**|DB 查询必须在子线程 (Room 默认检测)|
|**网络请求**|所有网络调用必须异步|
|**文件 I/O**|大文件读写使用 IO 线程|
|**复杂计算**|加密、图片处理等放在后台线程|
|**Message 堆积**|检查是否有无限 post Message 的循环|
|**BroadcastReceiver**|务必使用 `goAsync()` 做异步工作|
|**View 树深度**|避免超过 10 层，控制 onDraw 复杂度|
|**Lock 竞争**|避免主线程等待其他线程释放锁|

### **5. 最佳实践总结**

```kotlin
// 模板：安全的后台操作模式
class SafeOperationExample {
    
    // 方案 1：Coroutine + ViewModel（推荐）
    private val viewModel: MyViewModel by viewModels()
    
    fun loadData() {
        viewModel.loadDataAsync().observe(this) { data ->
            updateUI(data)
        }
    }
    
    // 方案 2：RxJava
    fun loadDataRx() {
        dataRepository.getData()
            .subscribeOn(Schedulers.io())
            .observeOn(AndroidSchedulers.mainThread())
            .subscribe { data -> updateUI(data) }
    }
    
    // 方案 3：Thread + Handler
    fun loadDataThread() {
        Thread {
            val data = expensiveOperation()
            mainHandler.post { updateUI(data) }
        }.start()
    }
    
    // 方案 4：Service（适合后台任务）
    startService(Intent(this, BackgroundService::class.java))
}
```

## **关键思路总结**

1. **ANR 根本原因**：主线程被阻塞超过阈值（5s~200s 取决于类型）
2. **检测机制**：InputDispatcher + ActivityManagerService 通过 Watchdog 检测
3. **解决方向**：
    - 立即异步化主线程上的耗时操作
    - 使用 Coroutine/RxJava/Thread 转移工作到子线程
    - 优化 View 绘制和消息处理
4. **预防方法**：
    - 遵守主线程职责（只处理 UI 和事件）
    - 代码审查和工具监控
    - 性能测试覆盖边界场景
